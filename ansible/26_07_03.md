# 2. Restore 자동화 목적

기존에는 Restic Snapshot을 복원한 뒤 대상별 복구 명령을 수동으로 실행해야 했다.

수동 복구 흐름:

```text
Backup Node에서 최신 Snapshot 복원
→ DB Dump 파일 확인
→ DB Primary로 파일 전달 및 Import
→ WordPress Archive 확인 및 Volume 복구
→ HAProxy Archive 확인 및 설정 적용
→ 서비스별 상태 검증
```

이번 작업에서는 이 절차를 Ansible Playbook으로 분리하였다.

목표 구조:

```text
Control Node에서 restore.yml 실행
→ Backup Node에서 Snapshot 준비
→ 선택한 복구 대상만 실행
→ 대상 서비스 복구
→ 서비스 상태 검증
```

# 3. Restore 대상 구분

복구 대상은 다음처럼 구분하였다.

| Restore Target | 대상 | 역할 |
|---|---|---|
| `db` | MariaDB Primary | WordPress DB Dump 복구 |
| `web` | Web1 Docker Swarm | WordPress 파일 및 Volume 복구 |
| `proxy` | HAProxy Proxy Node | HAProxy 설정 복구 |
| `monitoring` | Monitoring Node | Prometheus 설정 복구 |
| `all` | 전체 지원 대상 | 대상별 Restore 순차 실행 |

초기에는 `backup` 대상도 검토하였지만, Restore 폴더의 `backup.yml`은 Backup Node 자체 복구가 아니라 Restic Repository와 복구 파일을 다시 검증하는 역할이었다.

각 대상 Playbook에서 이미 필요한 파일 및 서비스 검증을 수행하므로 `backup` Restore Target은 제거하였다.

최종 허용 대상:

```yaml
restore_allowed_targets:
  - db
  - web
  - proxy
  - monitoring
  - all
```

# 7. prepare.yml 공통 복구 준비

파일:

```text
playbooks/restore/prepare.yml
```

`prepare.yml`은 실제 서비스별 복구 전에 Backup Node에서 공통으로 실행하는 작업이다.

주요 처리 흐름:

```text
Restore Target 값 검증
→ restore_confirm 승인값 확인
→ /backup Cinder Volume 마운트 확인
→ Restic Repository 접근 확인
→ Restore 작업 경로 초기화
→ 지정한 Snapshot을 restore-test 경로에 복원
→ 복원 결과 확인
```

추가로 Restic Repository 자체 무결성을 검사하도록 정리하였다.

```bash
restic check
```

환경변수:

```text
RESTIC_REPOSITORY=/backup/restic-repo
RESTIC_PASSWORD_FILE=/etc/restic/restic-password
RESTIC_CACHE_DIR=/backup/restic-cache
```

`restic check`는 DB, WordPress, HAProxy 파일 검증과 달리 Repository 자체의 Snapshot, Tree, Blob 구조를 검사하므로 공통 준비 단계에 두는 것이 적절하다고 판단하였다.

# 9. wordpress.yml WordPress 파일 복구

파일:

```text
playbooks/restore/wordpress.yml
```

복구 대상:

```text
Web1 Docker Swarm
WordPress Service: dandelion-wp_wordpress
WordPress Volume: dandelion_wordpress_data
```

복구 자료:

```text
wordpress-replica-1-volume.tar.gz
wordpress-replica-2-volume.tar.gz
```

복구 시 고려해야 할 항목:

```text
현재 WordPress Service 상태
Docker Volume 존재 여부
현재 Volume 안전 백업
복구 중 서비스 정지 또는 Replica 조정
Archive 압축 상태
복구 후 Replica 2/2 복귀
HTTP 응답 확인
```

작성한 Restore 흐름은 다음 방향으로 구성하였다.

```text
Backup Node에서 WordPress Archive 확인
→ Archive 압축 상태 확인
→ Web1로 Archive 직접 전달
→ 현재 WordPress Volume 안전 백업
→ Swarm Service 중지 또는 Replica 0 조정
→ Volume 데이터 복구
→ Service Replica 복구
→ Replica 2/2 및 HTTP 응답 확인
```

주의사항:

```text
현재 Web1은 단일 Docker Host에서 외부 Named Volume을 공유하는 구조이다.
복구 시 실제 Archive 내부 구조와 Volume 경로가 일치하는지 반드시 실행 전 확인해야 한다.
replica별 Archive 두 개의 데이터가 동일한지 또는 차이가 있는지 검증이 필요하다.
```

따라서 실제 실행 전 다음 항목을 우선 확인해야 한다.

```bash
tar -tzf wordpress-replica-1-volume.tar.gz | head
tar -tzf wordpress-replica-2-volume.tar.gz | head
```

# 10. proxy.yml HAProxy 설정 복구

파일:

```text
playbooks/restore/proxy.yml
```

복구 대상:

```text
Proxy Node
→ /etc/haproxy
```

복구 Archive:

```text
/backup/restore-test/backup/staging/haproxy/haproxy-config.tar.gz
```

주요 처리 흐름:

```text
Backup Node의 HAProxy Archive 확인
→ Archive 압축 상태 검사
→ Proxy Node로 Archive 직접 전달
→ 임시 경로에 압축 해제
→ 복구할 haproxy.cfg 존재 여부 확인
→ 현재 /etc/haproxy 안전 백업
→ 복구 후보 설정 문법 검사
→ /etc/haproxy에 설정 적용
→ 적용된 설정 재검사
→ HAProxy 서비스 재시작
→ HTTPS 응답 확인
→ 임시 파일 정리
```

HAProxy 설정은 잘못 적용하면 외부 접속이 즉시 중단될 수 있으므로 적용 전에 다음 검사를 수행하도록 구성하였다.

```bash
haproxy -c -f <복구할 haproxy.cfg>
```

적용 후에도 다시 검사한다.

```bash
haproxy -c -f /etc/haproxy/haproxy.cfg
```

주의사항:

```text
Archive 내부 경로가 etc/haproxy인지 haproxy인지 실제 파일 구조 확인 필요
SSL 인증서 파일이 Archive에 포함되는지 확인 필요
현재 인증서와 복구 인증서의 권한 및 유효성 확인 필요
```

# 13. Backup Node 자체 복구 범위 정리

Restore 폴더의 `backup.yml`은 Backup Node 자체를 복구하는 파일이 아니었다.

Backup Node 자체가 장애 나면 기존 구성 Playbook을 다시 사용한다.

```text
playbooks/backup.yml
```

Backup Node 인스턴스 장애 복구 흐름:

```text
새 Backup Node 생성
→ 기존 /backup Cinder Volume 연결
→ /backup 마운트
→ Ansible Inventory에 Backup Node 등록
→ common.yml 실행
→ backup.yml 실행
→ Restic Repository 접근 확인
→ Cron 및 백업 스크립트 상태 확인
```

실행 예시:

```bash
ansible-playbook playbooks/common.yml --limit backup-node
ansible-playbook playbooks/backup.yml
```

단, Backup Node의 Cinder Volume 자체가 유실되면 현재 `/backup/restic-repo`도 함께 유실된다.

현재 구조로는 Repository Volume 자체 장애까지 복구할 수 없다.

추후 필요한 항목:

```text
Cinder Volume Snapshot
Restic Repository 원격 복제
S3 또는 Object Storage 저장
별도 Backup Node 이중화
```

2026-07-03 작업에서는 기존 수동 Restore 시나리오를 기반으로 서비스별 Ansible Restore 자동화 구조를 작성하였다.

```text
Control Node

DB Primary
→ WordPress DB Import 및 테이블 검증

Web1 Docker Swarm
→ WordPress Volume 복구
→ Replica 2/2 및 HTTP 검증

Proxy Node
→ HAProxy 설정 복구
→ 문법 검사 및 HTTPS 검증

Monitoring Node
→ 실제 Prometheus 설정 백업 미구성으로 현재 복구 보류
```

```text
→ 완료

→ 미완료
```
