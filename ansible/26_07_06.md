# 3. 기존 Snapshot의 DB 백업 상태 확인

처음 `latest` Snapshot을 이용하여 MariaDB Restore를 실행하였다.

실행 형식:

```bash
ansible-playbook playbooks/restore.yml \
  -e restore_target=db \
  -e restore_snapshot=latest \
  -e restore_confirm=RESTORE
```

Restore 과정에서 다음 작업까지 진행되었다.

```text
WordPress Service Replica 0으로 중지
DB Dump 파일 전달
기존 DB 안전 백업
SQL Dump Import
복구 후 WordPress 테이블 수 검증
```

그러나 복구된 SQL Dump에 WordPress 테이블 생성문과 데이터 입력문이 없어서 테이블 수 검증에 실패하였다.

기존 Snapshot들을 조사한 결과:

```text
CREATE TABLE 수: 0
INSERT INTO 수: 0
```

즉 Restore Playbook 자체의 문제가 아니라, 당시 백업 시점의 `wordpress_db`가 비어 있었기 때문에 빈 SQL Dump가 Snapshot에 저장된 상태였다.

이 과정에서 DB에 기존 데이터가 없었던 것도 함께 확인하여 첫 Restore로 인한 실제 데이터 손실은 없었다.

# 4. WordPress 서비스 상태 복구 및 이미지 불일치 수정

첫 DB Restore 실패 후 WordPress Service가 Replica 0 상태로 내려가 있어 다시 Replica 2개로 복구하였다.

점검 과정에서 실행 중인 WordPress Service 이미지가 Ansible 설정과 다르게 동작하는 상태를 확인하였다.

```text
실행 중이던 이미지
→ wordpress:latest

Ansible에서 의도한 이미지
→ dandelion-wordpress:v1
```

`playbooks/wordpress.yml`을 다시 적용하여 실제 Swarm Service를 의도한 Custom Image로 수정하였다.

최종 상태:

```text
Service: dandelion-wp_wordpress
Image: dandelion-wordpress:v1
Replica: 2/2
```

이후 WordPress 초기 설치를 완료하고 실제 복구 검증용 데이터를 생성하였다.

```text

```

처음에는 DB Primary 보안그룹에 Web1의 TCP 3306을 직접 허용해야 하는 것으로 판단하였다.

## 7.2 실제 HAProxy DB 프록시 구성 확인

팀원이 Proxy Node에서 MariaDB TCP Proxy를 이미 구성한 사실을 확인하였다.



```text

```

## 7.4 Ansible Become Timeout 해결

DB Primary 상태 점검 중 다음 오류가 발생하였다.

```text
Timeout waiting for privilege escalation prompt
```

`ansible.cfg`에 전역 `become=True`가 설정되어 있어 일반 명령에도 권한 상승이 적용되는 상태였다.

먼저 `ansible_become=false`로 SSH 및 sudo 상태를 확인하였다.

```text
SSH 사용자: ubuntu
sudo -n true: 성공
```

이후 기존 SSH 연결을 초기화하였다.

```bash
ansible db-primary-node -m meta -a reset_connection \
  -e ansible_become=false
```

재확인 결과:

```text
whoami → root
```

sudo 설정 자체에는 문제가 없었고, 기존 SSH 연결의 권한 상승 세션이 일시적으로 꼬인 것으로 확인하였다.

# 11. HAProxy Restore 첫 실행 실패 및 수정

## 11.1 첫 실행 실패

```bash
ansible-playbook playbooks/restore.yml \
  -e restore_target=proxy \
  -e restore_snapshot=5d102766 \
  -e restore_confirm=RESTORE
```

Archive 전달과 압축 해제는 성공했지만 다음 단계에서 실패하였다.

```text
Archive 안에서 haproxy.cfg를 찾지 못했습니다.
```

실제 Archive 구조:

```text
haproxy/haproxy.cfg
```

기존 Restore Playbook 예상 구조:

```text
etc/haproxy/haproxy.cfg
```

즉 Archive 파일이 없거나 손상된 것이 아니라, `playbooks/restore/proxy.yml`이 압축 해제 후 잘못된 경로를 검사하고 있었다.

## 11.2 proxy.yml 경로 수정

파일:

```text
playbooks/restore/proxy.yml
```

기존 경로:

```text
{{ restore_haproxy_extract_dir }}/etc/haproxy/haproxy.cfg
{{ restore_haproxy_extract_dir }}/etc/haproxy/
```

수정 경로:

```text
{{ restore_haproxy_extract_dir }}/haproxy/haproxy.cfg
{{ restore_haproxy_extract_dir }}/haproxy/
```

수정 대상:

```text
복원된 haproxy.cfg 존재 여부 확인 경로
복구 후보 설정 문법 검사 경로
/etc/haproxy에 적용할 Copy 원본 경로
```

이전 실행은 실제 `/etc/haproxy` 적용 전에 실패했으므로 운영 HAProxy에는 영향이 없었다.

# 16. 발견된 문제와 해결 내용

| 문제 | 원인 | 해결 |
|---|---|---|
| 첫 DB Restore 후 테이블 0개 | 기존 Snapshot의 DB Dump가 빈 상태 | 실제 WordPress 데이터 생성 후 새 Snapshot 생성 |
| WordPress Service Health Check 실패 | WordPress가 DB Primary에 직접 접속하도록 설정됨 | 실제 구조에 맞게 Proxy 192.168.4.32:3306 사용 |
| DB Primary 권한 상승 Timeout | 기존 SSH 연결에서 Become 응답이 일시적으로 꼬임 | `reset_connection` 후 재접속 |
| 실행 이미지가 `wordpress:latest` | 실제 Service와 Ansible 의도 상태 불일치 | `playbooks/wordpress.yml` 재적용 |
| Proxy Restore에서 haproxy.cfg 미발견 | Archive 경로와 Playbook 예상 경로 불일치 | `etc/haproxy` 경로를 `haproxy` 경로로 수정 |
| Monitoring Restore 불가 | Monitoring 및 백업 자체 미구현 | Monitoring 구축 후 Backup/Restore 추가 예정 |
