# Phase 1 Ansible 자동화 작업 정리
## HAProxy 로드밸런서 구성부터 백업 자동화까지

## 1. 문서 목적

본 문서는 Team Dandelion 프로젝트에서 기존 `docker.yml`, `database.yml`, `wordpress.yml` 작성 이후 진행한 작업을 정리한다.

오늘 진행한 주요 작업은 다음과 같다.

```text
HAProxy 공통 변수 추가
→ haproxy.cfg Jinja2 템플릿 작성
→ proxy.yml 작성
→ HAProxy 코드 구조 및 실행 흐름 설명
→ backup.yml 설계 및 작성
→ MariaDB 및 WordPress 파일 백업 흐름 구성
→ 핵심 Playbook 작성 완료 여부 확인
```

현재 Playbook은 코드 작성 단계이며 실제 노드 적용과 실행 검증은 아직 진행하지 않았다.

변수별 역할:

| 변수 | 역할 |
|---|---|
| `haproxy_frontend_port` | 사용자의 요청을 받는 Proxy 노드 포트 |
| `haproxy_backend_port` | HAProxy가 Web 노드로 요청을 전달할 포트 |
| `haproxy_balance_algorithm` | Web 노드 요청 분산 방식 |

현재 연결 구조:

```text
사용자
→ Proxy Node:80
→ web1-node:80 또는 web2-node:80
```

# 4. HAProxy 설정 템플릿 작성

## 4.1 파일 위치

```text
~/dandeliondir/ansible/templates/haproxy.cfg.j2
```

## 4.2 작성 내용

```haproxy
# Ansible에서 자동 생성한 HAProxy 설정 파일

global
    log /dev/log local0
    log /dev/log local1 notice
    chroot /var/lib/haproxy
    stats socket /run/haproxy/admin.sock mode 660 level admin
    stats timeout 30s
    user haproxy
    group haproxy
    daemon

defaults
    log global
    mode http
    option httplog
    option dontlognull
    option forwardfor
    option http-server-close

    timeout connect 5s
    timeout client 50s
    timeout server 50s
    timeout http-request 10s
    timeout http-keep-alive 10s

frontend wordpress_frontend
    bind *:{{ haproxy_frontend_port }}
    default_backend wordpress_backend

backend wordpress_backend
    balance {{ haproxy_balance_algorithm }}
    option httpchk GET /

{% for host in groups['web'] %}
    server {{ host }} {{ hostvars[host].ansible_host }}:{{ haproxy_backend_port }} check inter 3s fall 3 rise 2
{% endfor %}
```

## 4.3 실제 변환 예상 결과

현재 Inventory와 변수값이 적용되면 backend 설정은 다음과 같이 생성된다.

```haproxy
backend wordpress_backend
    balance roundrobin
    option httpchk GET /

    server web1-node 192.168.4.4:80 check inter 3s fall 3 rise 2
    server web2-node 192.168.4.35:80 check inter 3s fall 3 rise 2
```

## 4.4 핵심 설정 설명

### frontend

```haproxy
frontend wordpress_frontend
    bind *:80
    default_backend wordpress_backend
```

Proxy 노드의 모든 네트워크 인터페이스에서 TCP 80번 포트로 요청을 받고 `wordpress_backend`로 전달한다.

### roundrobin

```haproxy
balance roundrobin
```

정상 상태인 Web 노드에 요청을 순서대로 분산한다.

```text
요청 1 → web1-node
요청 2 → web2-node
요청 3 → web1-node
요청 4 → web2-node
```

### Health Check

```haproxy
option httpchk GET /
```

HAProxy가 Web 노드의 `/` 경로에 HTTP 요청을 보내 상태를 확인한다.

```haproxy
check inter 3s fall 3 rise 2
```

| 옵션 | 의미 |
|---|---|
| `inter 3s` | 3초 간격으로 상태 확인 |
| `fall 3` | 3회 연속 실패하면 장애 처리 |
| `rise 2` | 2회 연속 성공하면 정상 상태로 복귀 |

### 접속자 IP 전달

```haproxy
option forwardfor
```

Web 서버에 `X-Forwarded-For` 헤더를 추가하여 실제 접속자 IP를 전달한다.

# 5. proxy.yml 작성

## 5.1 파일 위치

```text
~/dandeliondir/ansible/playbooks/proxy.yml
```

## 5.2 작성 내용

```yaml
---
- name: Proxy 노드 HAProxy 로드밸런서 구성
  hosts: proxy
  become: true
  gather_facts: true

  tasks:
    - name: Web 노드 그룹 구성 확인
      ansible.builtin.assert:
        that:
          - groups['web'] is defined
          - groups['web'] | length > 0
        fail_msg: "Inventory에 web 그룹 또는 Web 노드가 없습니다."
        success_msg: "HAProxy backend로 사용할 Web 노드를 확인했습니다."

    - name: HAProxy 패키지 설치
      ansible.builtin.apt:
        name:
          - haproxy
        state: present
        update_cache: true
        cache_valid_time: 3600

    - name: Proxy 노드에서 Web 노드 HTTP 응답 확인
      ansible.builtin.uri:
        url: "http://{{ hostvars[item].ansible_host }}:{{ haproxy_backend_port }}"
        status_code:
          - 200
          - 301
          - 302
        follow_redirects: none
        return_content: false
      loop: "{{ groups['web'] }}"
      register: web_backend_check
      retries: 5
      delay: 3
      until: web_backend_check.status in [200, 301, 302]

    - name: HAProxy 설정 파일 배포
      ansible.builtin.template:
        src: ../templates/haproxy.cfg.j2
        dest: /etc/haproxy/haproxy.cfg
        owner: root
        group: root
        mode: "0644"
        backup: true
        validate: "/usr/sbin/haproxy -c -f %s"
      notify: HAProxy 서비스 재시작

    - name: 변경된 HAProxy 설정 즉시 반영
      ansible.builtin.meta: flush_handlers

    - name: HAProxy 서비스 활성화 및 시작
      ansible.builtin.systemd_service:
        name: haproxy
        state: started
        enabled: true

    - name: HAProxy 포트 응답 대기
      ansible.builtin.wait_for:
        host: 127.0.0.1
        port: "{{ haproxy_frontend_port }}"
        timeout: 60

    - name: HAProxy 설정 문법 확인
      ansible.builtin.command:
        cmd: haproxy -c -f /etc/haproxy/haproxy.cfg
      register: haproxy_config_check
      changed_when: false

    - name: HAProxy 경유 WordPress HTTP 응답 확인
      ansible.builtin.uri:
        url: "http://127.0.0.1:{{ haproxy_frontend_port }}"
        status_code:
          - 200
          - 301
          - 302
        follow_redirects: none
        return_content: false
      register: haproxy_http_result
      retries: 10
      delay: 5
      until: haproxy_http_result.status in [200, 301, 302]

    - name: HAProxy 버전 확인
      ansible.builtin.command:
        cmd: haproxy -v
      register: haproxy_version_result
      changed_when: false

    - name: HAProxy 구성 결과 출력
      ansible.builtin.debug:
        msg:
          - "{{ haproxy_version_result.stdout_lines[0] }}"
          - "{{ haproxy_config_check.stdout }}"
          - "Proxy 노드: {{ inventory_hostname }}"
          - "접속 주소: http://{{ ansible_host }}:{{ haproxy_frontend_port }}"
          - "HTTP 상태 코드: {{ haproxy_http_result.status }}"

  handlers:
    - name: HAProxy 서비스 재시작
      ansible.builtin.systemd_service:
        name: haproxy
        state: restarted
```

# 6. proxy.yml 실행 흐름

```text
proxy 그룹 선택
→ Inventory의 Web 노드 확인
→ HAProxy 패키지 설치
→ Proxy 노드에서 Web1/Web2 HTTP 접속 확인
→ haproxy.cfg 템플릿 배포
→ 배포 전 설정 문법 검사
→ 설정 변경 시 HAProxy 즉시 재시작
→ HAProxy 서비스 실행 및 부팅 자동 시작
→ Proxy 노드의 80번 포트 확인
→ HAProxy 경유 WordPress HTTP 응답 확인
→ HAProxy 버전과 접속 결과 출력
```

## 6.1 Web 노드 그룹 확인

```yaml
groups['web'] is defined
groups['web'] | length > 0
```

Inventory에 `web` 그룹이 존재하고 Web 노드가 한 대 이상 등록되어 있는지 검사한다.

## 6.2 Proxy에서 Web 노드 직접 확인

```yaml
url: "http://{{ hostvars[item].ansible_host }}:{{ haproxy_backend_port }}"
loop: "{{ groups['web'] }}"
```

실제 확인 대상:

```text
http://192.168.4.4:80
http://192.168.4.35:80
```

HAProxy를 설정하기 전에 Proxy 노드와 Web 노드 사이의 네트워크 및 HTTP 연결을 확인한다.

## 6.3 설정 파일 사전 문법 검사

```yaml
validate: "/usr/sbin/haproxy -c -f %s"
```

새로운 설정 파일을 `/etc/haproxy/haproxy.cfg`에 반영하기 전에 HAProxy 문법을 검사한다.

문법 오류가 있으면 새 설정을 적용하지 않고 Playbook을 실패 처리한다.

## 6.4 Handler와 flush_handlers

```yaml
notify: HAProxy 서비스 재시작
```

`haproxy.cfg` 내용이 변경된 경우에만 재시작 Handler를 예약한다.

```yaml
ansible.builtin.meta: flush_handlers
```

이후 포트 및 HTTP 검증 전에 설정을 즉시 반영하기 위해 예약된 Handler를 실행한다.

## 6.5 HAProxy 경유 응답 확인

```text
Proxy Node 127.0.0.1:80
→ HAProxy frontend
→ wordpress_backend
→ web1-node 또는 web2-node
→ WordPress 응답
```

HTTP 상태 코드 `200`, `301`, `302`를 정상으로 처리한다.

이 방식은 현재 Control 노드에서 모든 관리 노드로 접속 가능한 Ansible 구조를 그대로 이용한다.

# 10. backup.yml 작성

## 10.1 파일 위치

```text
~/dandeliondir/ansible/playbooks/backup.yml
```

## 10.2 Play 구성

`backup.yml`은 총 다섯 개의 Play로 구성하였다.

```text
1. Control 노드에서 백업 실행 ID 생성
2. DB 노드에서 MariaDB 백업 생성
3. Web 노드에서 WordPress 파일 백업 생성
4. Backup 노드에 모든 백업 파일 저장 및 검증
5. Control 노드 임시 파일 정리
```

## 10.3 작성 내용

```yaml
---
- name: 백업 실행 정보 준비
  hosts: localhost
  gather_facts: false
  become: false

  tasks:
    - name: 백업 실행 식별자 생성
      ansible.builtin.set_fact:
        dandelion_backup_id: "{{ lookup('pipe', 'date +%Y%m%d_%H%M%S') }}"

    - name: Control 노드 임시 백업 디렉터리 생성
      ansible.builtin.file:
        path: "{{ backup_control_staging_root }}/{{ dandelion_backup_id }}"
        state: directory
        mode: "0700"


- name: MariaDB 데이터베이스 백업
  hosts: db
  become: true
  gather_facts: false

  vars:
    backup_id: "{{ hostvars['localhost'].dandelion_backup_id }}"
    control_staging_dir: "{{ backup_control_staging_root }}/{{ backup_id }}"
    database_backup_filename: >-
      wordpress-db-{{ inventory_hostname }}-{{ backup_id }}.sql.gz
    database_backup_path: >-
      {{ backup_remote_work_dir }}/{{ database_backup_filename }}

  tasks:
    - name: MariaDB 백업 도구 설치
      ansible.builtin.apt:
        name:
          - gzip
        state: present
        update_cache: true
        cache_valid_time: 3600

    - name: DB 노드 임시 백업 디렉터리 생성
      ansible.builtin.file:
        path: "{{ backup_remote_work_dir }}"
        state: directory
        owner: root
        group: root
        mode: "0750"

    - name: WordPress 데이터베이스 덤프 생성
      ansible.builtin.shell:
        cmd: |
          set -o pipefail
          umask 077

          mariadb-dump \
            --user=root \
            --protocol=socket \
            --socket=/run/mysqld/mysqld.sock \
            --single-transaction \
            --quick \
            --skip-lock-tables \
            --routines \
            --triggers \
            --events \
            --default-character-set=utf8mb4 \
            "{{ wordpress_db_name }}" \
            | gzip -9 > "{{ database_backup_path }}"
        executable: /bin/bash
      changed_when: true

    - name: MariaDB 백업 파일 정보 확인
      ansible.builtin.stat:
        path: "{{ database_backup_path }}"
      register: database_backup_stat

    - name: MariaDB 백업 파일 생성 결과 검증
      ansible.builtin.assert:
        that:
          - database_backup_stat.stat.exists
          - database_backup_stat.stat.isreg
          - database_backup_stat.stat.size | int > 0
        fail_msg: "MariaDB 백업 파일이 정상적으로 생성되지 않았습니다."
        success_msg: "MariaDB 백업 파일이 정상적으로 생성되었습니다."

    - name: MariaDB 백업 파일 소유권 변경
      ansible.builtin.file:
        path: "{{ database_backup_path }}"
        owner: ubuntu
        group: ubuntu
        mode: "0600"

    - name: MariaDB 백업 파일을 Control 노드로 가져오기
      ansible.builtin.fetch:
        src: "{{ database_backup_path }}"
        dest: "{{ control_staging_dir }}/"
        flat: true
        validate_checksum: true
      become: false

    - name: DB 노드 임시 백업 파일 제거
      ansible.builtin.file:
        path: "{{ database_backup_path }}"
        state: absent


- name: WordPress 파일 백업
  hosts: web
  become: true
  gather_facts: false

  vars:
    backup_id: "{{ hostvars['localhost'].dandelion_backup_id }}"
    control_staging_dir: "{{ backup_control_staging_root }}/{{ backup_id }}"
    wordpress_backup_filename: >-
      wordpress-files-{{ inventory_hostname }}-{{ backup_id }}.tar.gz
    wordpress_backup_path: >-
      {{ backup_remote_work_dir }}/{{ wordpress_backup_filename }}

  tasks:
    - name: WordPress 백업 도구 설치
      ansible.builtin.apt:
        name:
          - gzip
        state: present
        update_cache: true
        cache_valid_time: 3600

    - name: Web 노드 임시 백업 디렉터리 생성
      ansible.builtin.file:
        path: "{{ backup_remote_work_dir }}"
        state: directory
        owner: root
        group: root
        mode: "0750"

    - name: WordPress 컨테이너 정보 확인
      community.docker.docker_container_info:
        name: "{{ wordpress_container_name }}"
      register: wordpress_container_info

    - name: WordPress 컨테이너 실행 상태 검증
      ansible.builtin.assert:
        that:
          - wordpress_container_info.exists
          - wordpress_container_info.container.State.Running | default(false) | bool
        fail_msg: >-
          {{ wordpress_container_name }} 컨테이너가 없거나 실행 중이 아닙니다.
        success_msg: "WordPress 컨테이너가 실행 중입니다."

    - name: WordPress 파일 압축 백업 생성
      ansible.builtin.shell:
        cmd: |
          set -o pipefail
          umask 077

          docker exec "{{ wordpress_container_name }}" \
            tar -cf - -C /var/www/html . \
            | gzip -9 > "{{ wordpress_backup_path }}"
        executable: /bin/bash
      changed_when: true

    - name: WordPress 백업 파일 정보 확인
      ansible.builtin.stat:
        path: "{{ wordpress_backup_path }}"
      register: wordpress_backup_stat

    - name: WordPress 백업 파일 생성 결과 검증
      ansible.builtin.assert:
        that:
          - wordpress_backup_stat.stat.exists
          - wordpress_backup_stat.stat.isreg
          - wordpress_backup_stat.stat.size | int > 0
        fail_msg: "WordPress 파일 백업이 정상적으로 생성되지 않았습니다."
        success_msg: "WordPress 파일 백업이 정상적으로 생성되었습니다."

    - name: WordPress 백업 파일 소유권 변경
      ansible.builtin.file:
        path: "{{ wordpress_backup_path }}"
        owner: ubuntu
        group: ubuntu
        mode: "0600"

    - name: WordPress 백업 파일을 Control 노드로 가져오기
      ansible.builtin.fetch:
        src: "{{ wordpress_backup_path }}"
        dest: "{{ control_staging_dir }}/"
        flat: true
        validate_checksum: true
      become: false

    - name: Web 노드 임시 백업 파일 제거
      ansible.builtin.file:
        path: "{{ wordpress_backup_path }}"
        state: absent


- name: Backup 노드에 백업 파일 저장
  hosts: backup
  become: true
  gather_facts: true

  vars:
    backup_id: "{{ hostvars['localhost'].dandelion_backup_id }}"
    control_staging_dir: "{{ backup_control_staging_root }}/{{ backup_id }}"
    backup_destination_dir: "{{ backup_storage_dir }}/{{ backup_id }}"

  tasks:
    - name: Backup 노드 기본 저장 디렉터리 생성
      ansible.builtin.file:
        path: "{{ backup_storage_dir }}"
        state: directory
        owner: root
        group: root
        mode: "0750"

    - name: 현재 실행 백업 디렉터리 생성
      ansible.builtin.file:
        path: "{{ backup_destination_dir }}"
        state: directory
        owner: root
        group: root
        mode: "0750"

    - name: MariaDB 백업 파일을 Backup 노드에 저장
      ansible.builtin.copy:
        src: >-
          {{ control_staging_dir }}/wordpress-db-{{ item }}-{{ backup_id }}.sql.gz
        dest: >-
          {{ backup_destination_dir }}/wordpress-db-{{ item }}-{{ backup_id }}.sql.gz
        owner: root
        group: root
        mode: "0600"
      loop: "{{ groups['db'] }}"

    - name: WordPress 백업 파일을 Backup 노드에 저장
      ansible.builtin.copy:
        src: >-
          {{ control_staging_dir }}/wordpress-files-{{ item }}-{{ backup_id }}.tar.gz
        dest: >-
          {{ backup_destination_dir }}/wordpress-files-{{ item }}-{{ backup_id }}.tar.gz
        owner: root
        group: root
        mode: "0600"
      loop: "{{ groups['web'] }}"

    - name: Backup 노드에 저장된 백업 파일 검색
      ansible.builtin.find:
        paths: "{{ backup_destination_dir }}"
        file_type: file
        recurse: false
        patterns:
          - "*.sql.gz"
          - "*.tar.gz"
      register: stored_backup_files

    - name: 백업 파일 개수 검증
      ansible.builtin.assert:
        that:
          - >-
            stored_backup_files.matched | int ==
            (groups['db'] | length) + (groups['web'] | length)
        fail_msg: "Backup 노드에 필요한 백업 파일이 모두 저장되지 않았습니다."
        success_msg: "DB 및 Web 노드 백업 파일이 모두 저장되었습니다."

    - name: 백업 파일 SHA256 체크섬 계산
      ansible.builtin.command:
        argv:
          - sha256sum
          - "{{ item.path | basename }}"
        chdir: "{{ backup_destination_dir }}"
      loop: "{{ stored_backup_files.files }}"
      register: backup_checksum_results
      changed_when: false
      loop_control:
        label: "{{ item.path | basename }}"

    - name: SHA256 체크섬 파일 생성
      ansible.builtin.copy:
        dest: "{{ backup_destination_dir }}/SHA256SUMS"
        owner: root
        group: root
        mode: "0600"
        content: |
          # Dandelion Backup
          # Backup ID: {{ backup_id }}
          # Created: {{ ansible_date_time.iso8601 }}

          {% for result in backup_checksum_results.results %}
          {{ result.stdout }}
          {% endfor %}

    - name: 보관 기간이 지난 백업 디렉터리 검색
      ansible.builtin.find:
        paths: "{{ backup_storage_dir }}"
        file_type: directory
        recurse: false
        patterns:
          - "20*"
        age: "{{ backup_retention_days }}d"
      register: expired_backup_directories

    - name: 보관 기간이 지난 백업 디렉터리 제거
      ansible.builtin.file:
        path: "{{ item.path }}"
        state: absent
      loop: "{{ expired_backup_directories.files }}"
      loop_control:
        label: "{{ item.path | basename }}"

    - name: 백업 완료 상태 기록
      ansible.builtin.set_fact:
        dandelion_backup_complete: true

    - name: 백업 결과 출력
      ansible.builtin.debug:
        msg:
          - "백업 ID: {{ backup_id }}"
          - "저장 위치: {{ backup_destination_dir }}"
          - "백업 파일 수: {{ stored_backup_files.matched }}"
          - "보관 기간: {{ backup_retention_days }}일"


- name: Control 노드 임시 백업 파일 정리
  hosts: localhost
  gather_facts: false
  become: false

  vars:
    backup_id: "{{ hostvars['localhost'].dandelion_backup_id }}"
    control_staging_dir: "{{ backup_control_staging_root }}/{{ backup_id }}"

  tasks:
    - name: 백업 전달 완료 후 임시 디렉터리 제거
      ansible.builtin.file:
        path: "{{ control_staging_dir }}"
        state: absent
      when:
        - groups['backup'] | length > 0
        - >-
          hostvars[groups['backup'][0]].dandelion_backup_complete
          | default(false)
          | bool
```

# 11. backup.yml 주요 코드 설명

## 11.1 백업 실행 ID

```yaml
dandelion_backup_id: "{{ lookup('pipe', 'date +%Y%m%d_%H%M%S') }}"
```

Playbook 실행 시각을 기준으로 고유 백업 ID를 생성한다.

예시:

```text
20260624_153000
```

모든 노드의 백업 파일이 같은 ID를 사용하므로 한 번의 백업 실행 결과를 구분할 수 있다.

## 11.2 MariaDB 덤프 생성

```bash
mariadb-dump
```

주요 옵션:

| 옵션 | 역할 |
|---|---|
| `--single-transaction` | InnoDB 테이블을 장시간 잠그지 않고 일관된 덤프 생성 |
| `--quick` | 테이블 데이터를 한 번에 메모리에 올리지 않고 순차 처리 |
| `--skip-lock-tables` | 명시적인 테이블 잠금 방지 |
| `--routines` | Stored Procedure와 Function 포함 |
| `--triggers` | Trigger 포함 |
| `--events` | Event Scheduler 항목 포함 |
| `--default-character-set=utf8mb4` | WordPress 문자셋 지정 |

생성 결과:

```text
wordpress-db-db-node-백업ID.sql.gz
```

## 11.3 WordPress 파일 백업

먼저 WordPress 컨테이너가 존재하고 실행 중인지 확인한다.

```yaml
community.docker.docker_container_info:
  name: "{{ wordpress_container_name }}"
```

이후 컨테이너 내부의 `/var/www/html`을 압축한다.

```bash
docker exec dandelion-wordpress \
  tar -cf - -C /var/www/html . \
  | gzip -9 > wordpress-files.tar.gz
```

Web 노드별 생성 결과:

```text
wordpress-files-web1-node-백업ID.tar.gz
wordpress-files-web2-node-백업ID.tar.gz
```

## 11.4 fetch와 copy를 이용한 전달

```yaml
ansible.builtin.fetch
```

DB 및 Web 노드의 백업 파일을 Control 노드로 가져온다.

```yaml
ansible.builtin.copy
```

Control 노드의 임시 파일을 Backup 노드로 전송한다.

## 11.5 백업 파일 개수 검증

현재 DB 노드 1대와 Web 노드 2대를 기준으로 총 세 개의 백업 파일이 생성되어야 한다.

```text
DB 백업 파일 1개
WordPress 파일 백업 2개
총 3개
```

예상 개수와 실제 개수가 다르면 Playbook을 실패 처리한다.

## 11.6 SHA256 체크섬

각 백업 파일의 SHA256 값을 계산하여 다음 파일에 저장한다.

```text
SHA256SUMS
```

백업 파일이 전송 또는 보관 과정에서 손상되지 않았는지 검증할 때 사용한다.

## 11.7 오래된 백업 제거

```yaml
backup_retention_days: 7
```

현재 설정 기준으로 7일이 지난 백업 디렉터리를 검색하여 제거한다.

## 11.8 임시 파일 정리

Backup 노드 저장과 검증이 모두 성공한 경우에만 Control 노드의 임시 백업 디렉터리를 제거한다.

백업 실패 시에는 문제 확인을 위해 임시 파일이 남을 수 있다.

## 18.4 보안그룹

필요한 통신:


DB의 TCP 3306 포트는 인터넷 전체에 공개하지 않고 Web 노드에서만 접근하도록 제한한다.

```bash
ansible proxy -b -m shell -a "ss -lntp | grep ':80' || true"
```
