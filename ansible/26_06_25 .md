## 5.1 오류 발생

```bash

```

오류:

```text
ERROR! couldn't resolve module/action 'ansible.mysql.mysql_user'
```

오류가 발생한 Task:

```yaml
- name: MariaDB 익명 사용자 제거
  ansible.mysql.mysql_user:
```

## 5.2 원인

`database.yml`에서 사용한 다음 모듈은 Ansible 기본 내장 모듈이 아니라 `ansible.mysql` Collection에 포함되어 있다.

```text
ansible.mysql.mysql_user
ansible.mysql.mysql_db
```

Control 노드에 `ansible.mysql` Collection이 설치되어 있지 않아 모듈을 찾지 못한 것이 원인이었다.

# 10. WordPress Compose 템플릿 오류

## 10.1 오류 내용

처음 Web1에 `wordpress.yml`을 실행했을 때 Compose 파일의 최상위에 Ansible 변수명이 그대로 들어가 Docker Compose 검증에 실패하였다.

문제가 된 항목:

```text
wordpress_base_image
wordpress_image_name
wordpress_image_tag
wordpress_container_name
wordpress_host_port
wordpress_data_volume
```

이 변수들은 `group_vars/all.yml`에서 정의하고 Compose 템플릿에서는 값으로 참조해야 한다.

## 10.2 템플릿 수정

`templates/wordpress-compose.yml.j2`를 다음 구조로 수정하였다.

```yaml
---
services:
  wordpress:
    build:
      context: .
      args:
        WORDPRESS_BASE_IMAGE: "{{ wordpress_base_image }}"

    image: "{{ wordpress_image_name }}:{{ wordpress_image_tag }}"
    container_name: "{{ wordpress_container_name }}"
    restart: unless-stopped

    ports:
      - "{{ wordpress_host_port }}:80"

    environment:
      WORDPRESS_DB_HOST: "{{ hostvars[groups['db'][0]].ansible_host }}:{{ mariadb_port }}"
      WORDPRESS_DB_NAME: "{{ wordpress_db_name }}"
      WORDPRESS_DB_USER: "{{ wordpress_db_user }}"
      WORDPRESS_DB_PASSWORD: "{{ wordpress_db_password }}"

    volumes:
      - wordpress_data:/var/www/html

    healthcheck:
      test:
        - CMD-SHELL
        - curl -fsS http://127.0.0.1/ || exit 1
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 60s

volumes:
  wordpress_data:
    name: "{{ wordpress_data_volume }}"
```

수정 후 Compose 설정 검증 단계는 정상 통과하였다.

# 12. Web1 80번 포트 충돌

## 12.1 최초 오류

새 컨테이너 시작 단계에서 다음 오류가 발생하였다.

```text
Bind for 0.0.0.0:80 failed: port is already allocated
```

이후 재실행에서는 IPv6 주소에서도 동일한 충돌이 확인되었다.

```text
Bind for :::80 failed: port is already allocated
```

새 컨테이너 상태:

```text
dandelion-wordpress:v1 이미지 생성 완료
dandelion-wordpress 컨테이너 생성
컨테이너 상태: Created
호스트 포트 연결 실패
```

## 12.2 포트 점유 확인

```bash
ansible web1-node -b -m shell -a "ss -lntp '( sport = :80 )' || true; fuser -v 80/tcp 2>/dev/null || true"
```

결과:

```text
0.0.0.0:80  → docker-proxy
[::]:80     → docker-proxy
```

Docker 재시작 후 PID가 바뀌면서 `docker-proxy`가 다시 생성되었다.

이는 단순 잔여 프로세스가 아니라 재시작 정책이 설정된 기존 컨테이너가 Docker 서비스와 함께 다시 실행되고 있다는 의미였다.

## 12.3 실제 점유 컨테이너 확인

```bash
ansible web1-node -b -m command -a "docker ps --filter publish=80"
```

결과:

```text
컨테이너 이름: dandelion-wp1
이미지: dandelion-wordpress:2.0
포트: 0.0.0.0:80->80/tcp, [::]:80->80/tcp
```

처음 예상했던 기존 컨테이너 이름과 실제 실행 중인 이름이 달랐다.

```text
예상: dandelion-wp
실제: dandelion-wp1
```

# 17. 작업 중 확인한 핵심 사항

## 17.1 수동 구성과 Playbook이 함께 존재할 때

기존에 수동으로 실행한 컨테이너와 Playbook이 새로 생성하는 컨테이너가 같은 호스트 포트를 사용하면 충돌한다.

```text
기존 컨테이너: 호스트 80번 사용
새 Compose 컨테이너: 호스트 80번 사용
결과: port is already allocated
```

Playbook 실행 전 기존 컨테이너 이름, 포트, 재시작 정책을 확인해야 한다.

## 17.2 `docker ps -a`만으로는 원인 판단이 부족할 수 있음

80번 포트 충돌 확인에는 다음 명령을 함께 사용하는 것이 정확하다.

```bash
docker ps --filter publish=80
ss -lntp '( sport = :80 )'
fuser -v 80/tcp
```

## 17.3 Docker 재시작 정책 주의

기존 컨테이너에 `always` 또는 `unless-stopped` 정책이 설정되어 있으면 Docker 서비스를 재시작했을 때 컨테이너가 다시 실행되어 포트를 다시 점유할 수 있다.

전환 작업 시 다음 순서가 필요하다.

```text
docker update --restart=no <기존 컨테이너>
→ docker stop <기존 컨테이너>
→ 포트 해제 확인
→ 새 Playbook 실행
```

## 17.4 Ansible 문법 검사와 Docker Compose 검사는 다름

```bash
ansible-playbook --syntax-check
```

이 명령은 Ansible Playbook의 YAML 및 모듈 구조를 검사한다.

그러나 템플릿으로 생성되는 Docker Compose 파일의 유효성까지 완전히 검사하지는 않는다.

Compose 파일은 대상 노드에서 다음과 같이 별도 검증할 수 있다.

```bash
cd /opt/dandelion/wordpress
docker compose -f compose.yml config
```

## 17.5 현재 비밀번호 관리

현재 실습 환경에서는 다음 값이 평문 변수로 저장되어 있다.

```yaml
wordpress_db_password: "test123"
```

운영 환경에서는 Ansible Vault 또는 Docker Secret과 같은 보안 방식으로 변경해야 한다.

# 18. 최종 진행 상태

| Playbook | 상태 |
|---|---|

