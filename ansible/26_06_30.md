# 2026-06-30 Docker Swarm 구조 변경 및 Ansible 자동화 정리

## 1. 문서 범위

본 문서는 `26_06_26.md` 이후 진행한 작업을 정리한다.

기존 구조는 Web 인스턴스 두 대에 각각 WordPress 컨테이너를 배포하고 HAProxy가 두 Web 노드 사이에서 Round Robin 로드밸런싱을 수행하는 방식이었다.

이번 작업에서는 프로젝트 계획을 변경하여 다음 구조로 전환하기로 하였다.

```text
기존 구조
Client
→ HAProxy Load Balancer
→ Web1 WordPress
→ Web2 WordPress
→ MariaDB

변경 구조
Client
→ HAProxy Reverse Proxy
→ Web1 Docker Swarm Published Port
→ WordPress Replica 1 또는 Replica 2
→ MariaDB Primary
```

핵심 변경 내용:

```text
Web2 인스턴스 삭제
→ Web1 단일 인스턴스에 WordPress Replica 2개 배치
→ Docker Swarm이 Replica 유지 및 요청 분산 담당
→ HAProxy는 SSL 종료와 Reverse Proxy 역할만 담당
→ Ansible Playbook도 처음부터 Swarm을 사용하는 구조로 수정
```

# 2. 구조 변경 배경

기존에는 WordPress Web 서버 두 개를 각각 별도 OpenStack 인스턴스로 운영하였다.

```text
Web1 Instance
└── WordPress Container

Web2 Instance
└── WordPress Container
```

이 구조는 Web 인스턴스마다 Ubuntu 운영체제와 Docker Engine이 별도로 필요하여 프로젝트 실습 환경에서 자원 사용량이 커졌다.

이에 따라 Web 인스턴스를 한 대로 줄이고 그 안에서 WordPress 컨테이너 두 개를 Docker Swarm Replica로 실행하도록 변경하였다.

변경 후 기대 효과:

```text
Web 인스턴스 2대 → 1대
VM 운영체제 오버헤드 감소
WordPress Replica 수는 2개 유지
컨테이너 장애 시 Swarm이 Replica 재생성
Swarm Ingress를 통한 요청 분산
```

단, 단일 Swarm 노드 구조이므로 Web1 인스턴스 자체가 중단되면 WordPress Replica 두 개가 모두 중단된다.

따라서 이번 구성은 다음 범위를 목표로 한다.

```text
컨테이너 단위 장애 복구
Replica 수 유지
컨테이너 간 요청 분산

제외 범위
Web 노드 자체 장애에 대한 고가용성
다중 Manager 구성
다중 Worker 노드 분산
```

# 6. Docker Swarm 수동 초기화

처음에는 Control Node에서 직접 다음 명령을 실행하였다.

```bash
docker swarm init --advertise-addr 192.168.4.4
```

Control Node에는 Docker가 설치되어 있지 않아 다음 오류가 발생하였다.

```text
Command 'docker' not found
```

원인:

```text
명령 실행 위치: Control Node
실제 Swarm 구성 대상: Web1 Node
```

Control Node에서 Ansible을 통해 Web1에 명령을 전달하였다.

```bash
ansible web1-node -b -m command -a \
"docker swarm init --advertise-addr 192.168.4.4"
```

명령 의미:

```text
docker swarm init
→ 현재 Docker Engine을 Swarm Manager로 초기화

--advertise-addr 192.168.4.4
→ 다른 Swarm 노드가 Manager를 찾을 때 사용할 Web1 내부 IP
```

현재 구조는 단일 노드 Swarm이므로 Web1은 동시에 다음 역할을 수행한다.

```text
Swarm Manager
Swarm Worker
WordPress 실행 노드
```

# 11. swarm.yml 작성

파일 위치:

```text
~/dandeliondir/ansible/playbooks/swarm.yml
```

작성 내용:

```yaml
---
- name: Web 노드 Docker Swarm 환경 구성
  hosts: web
  become: true
  gather_facts: true

  tasks:
    - name: Docker Swarm Manager 상태 보장
      community.docker.docker_swarm:
        state: present
        advertise_addr: "{{ ansible_host }}"

    - name: Docker Swarm 정보 확인
      community.docker.docker_swarm_info:
        nodes: true
      register: swarm_info_result

    - name: Docker Swarm Manager 상태 검증
      ansible.builtin.assert:
        that:
          - swarm_info_result.can_talk_to_docker | bool
          - swarm_info_result.docker_swarm_active | bool
          - swarm_info_result.docker_swarm_manager | bool
        fail_msg: "Docker Swarm Manager 구성이 정상적이지 않습니다."
        success_msg: "Docker Swarm Manager 구성이 정상적으로 완료되었습니다."

    - name: Docker Swarm 노드 정보 출력
      ansible.builtin.debug:
        var: swarm_info_result.nodes
```

주요 동작:

```text
새 Web 노드
→ Swarm Manager 초기화

이미 Swarm이 활성화된 Web 노드
→ 현재 상태 유지

검증
→ Docker 데몬 통신 여부 확인
→ Swarm 활성 상태 확인
→ 현재 노드가 Manager인지 확인
→ Swarm 노드 목록 출력
```

`advertise_addr`는 Inventory의 `ansible_host`를 사용한다.

현재 실제 값:

```text
192.168.4.4
```

# 13. Docker Swarm PDF 검토

팀에서 제공한 `docker-swarm.pdf`와 현재 작업 방향을 비교하였다.

PDF의 작업 흐름:

```text
1. docker swarm init --advertise-addr 192.168.4.4
2. docker-compose.yml 작성
3. docker stack deploy 실행
4. HAProxy 설정 수정
5. docker service logs로 분산 확인
```

현재 Ansible 작업 흐름:

```text
1. swarm.yml로 Swarm Manager 상태 보장
2. wordpress-stack.yml.j2 작성
3. wordpress.yml에서 Stack 배포
4. proxy.yml을 Reverse Proxy 방식으로 수정
5. Service Replica와 로그를 통해 분산 확인
```

PDF와 현재 구조는 동일한 방향이다.

차이점:

```text
PDF
→ 수동 명령 기반

현재 프로젝트
→ Ansible Playbook 기반 자동화
```

PDF 예시의 다음 설정은 Swarm에서 사용하지 않기로 하였다.

```yaml
container_name: dandelion-wp
```

Swarm은 Task 이름을 자동으로 생성한다.

예상 형태:

```text
dandelion-wp_wordpress.1.<task-id>
dandelion-wp_wordpress.2.<task-id>
```

또한 Stack 파일 내부에서는 `build:`를 사용하지 않고, `wordpress.yml`에서 이미지를 먼저 생성한 뒤 Stack에서는 완성된 이미지를 참조하도록 구성한다.

# 14. Swarm 공통 변수 추가

파일 위치:

```text
~/dandeliondir/ansible/group_vars/all.yml
```

추가한 변수:

```yaml
# Docker Swarm WordPress 설정
wordpress_stack_name: "dandelion-wp"
wordpress_service_name: "wordpress"
wordpress_replicas: 2
wordpress_published_port: 80
wordpress_target_port: 80
wordpress_swarm_dir: "/opt/dandelion/wordpress"
```

변수 의미:

| 변수 | 값 | 역할 |
|---|---:|---|
| `wordpress_stack_name` | `dandelion-wp` | Docker Stack 이름 |
| `wordpress_service_name` | `wordpress` | Stack 내부 WordPress 서비스 이름 |
| `wordpress_replicas` | `2` | 유지할 WordPress Task 수 |
| `wordpress_published_port` | `80` | Web1에서 공개할 포트 |
| `wordpress_target_port` | `80` | WordPress 컨테이너 내부 포트 |
| `wordpress_swarm_dir` | `/opt/dandelion/wordpress` | Stack 및 이미지 빌드 파일 저장 위치 |

Stack 배포 후 생성될 서비스 이름:

```text
dandelion-wp_wordpress
```

기존 변수 중 다음 항목은 아직 삭제하지 않았다.

```yaml
wordpress_container_name
wordpress_host_port
wordpress_data_volume
```

`wordpress_data_volume`은 Swarm Stack에서도 사용한다.

`wordpress_container_name`, `wordpress_host_port`는 `backup.yml`, 기존 Playbook 정리가 끝난 후 제거 여부를 결정한다.

# 15. WordPress Stack 템플릿 작성

파일 위치:

```text
~/dandeliondir/ansible/templates/wordpress-stack.yml.j2
```

작성한 템플릿:

```yaml
---
version: "3.8"

services:
  wordpress:
    image: "{{ wordpress_image_name }}:{{ wordpress_image_tag }}"

    ports:
      - target: {{ wordpress_target_port }}
        published: {{ wordpress_published_port }}
        protocol: tcp
        mode: ingress

    environment:
      WORDPRESS_DB_HOST: "{{ hostvars[groups['db'][0]].ansible_host }}:{{ mariadb_port }}"
      WORDPRESS_DB_NAME: "{{ wordpress_db_name }}"
      WORDPRESS_DB_USER: "{{ wordpress_db_user }}"
      WORDPRESS_DB_PASSWORD: "{{ wordpress_db_password }}"

    volumes:
      - wordpress_data:/var/www/html

    healthcheck:
      test:
        - CMD-SHELL
        - curl -fsS http://127.0.0.1/ || exit 1
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 60s

    deploy:
      mode: replicated
      replicas: {{ wordpress_replicas }}

      restart_policy:
        condition: on-failure
        delay: 5s

      update_config:
        parallelism: 1
        delay: 10s
        order: start-first

volumes:
  wordpress_data:
    external: true
    name: "{{ wordpress_data_volume }}"
```

## 16.2 Swarm Ingress Port

```yaml
ports:
  - target: 80
    published: 80
    protocol: tcp
    mode: ingress
```

요청 흐름:

```text
HAProxy
→ Web1 192.168.4.4:80
→ Swarm Ingress
→ WordPress Replica 1 또는 Replica 2
```

## 16.3 DB 연결

```yaml
WORDPRESS_DB_HOST: "{{ hostvars[groups['db'][0]].ansible_host }}:{{ mariadb_port }}"
```

현재 Inventory 기준 실제 값:

```text
192.168.4.195:3306
```

WordPress Replica 두 개가 동일한 MariaDB Primary를 사용한다.

## 16.4 WordPress 공유 볼륨

```yaml
volumes:
  - wordpress_data:/var/www/html
```

외부 볼륨 이름:

```text
dandelion_wordpress_data
```

Web1 단일 Docker Host 안에서 두 Replica가 같은 WordPress 파일 볼륨을 공유한다.

## 16.5 Replica 수

```yaml
deploy:
  mode: replicated
  replicas: 2
```

Swarm은 WordPress Task 두 개가 실행 중인 상태를 유지한다.

한 Task가 중지되면 새로운 Task를 생성하여 다시 `2/2` 상태로 맞춘다.

## 16.6 재시작 정책

```yaml
restart_policy:
  condition: on-failure
  delay: 5s
```

Task가 실패하면 5초 후 Swarm이 다시 시작한다.

## 16.7 순차 업데이트

```yaml
update_config:
  parallelism: 1
  delay: 10s
  order: start-first
```

이미지 또는 서비스 설정 변경 시 Replica를 한 개씩 순차적으로 교체한다.

# 17. 기존 Compose 템플릿과 차이

기존 `wordpress-compose.yml.j2`에는 다음 설정이 있었다.

```yaml
build:
container_name:
restart: unless-stopped
```

Swarm Stack에서는 다음처럼 변경한다.

```text
build
→ wordpress.yml에서 별도 이미지 빌드

container_name
→ Swarm이 Task 이름 자동 관리

restart
→ deploy.restart_policy

일반 ports
→ Swarm ingress published port

컨테이너 1개
→ deploy.replicas: 2
```

기존 Compose 템플릿은 더 이상 최종 배포용으로 사용하지 않을 예정이다.

다만 현재는 롤백과 비교를 위해 즉시 삭제하지 않았다.

# 18. HAProxy 최종 구조 방향

기존 HAProxy는 다음 두 Backend로 요청을 분산하였다.

```text
192.168.4.4:80
192.168.4.35:80
```

변경 후 Backend는 다음 하나만 사용한다.

```text
192.168.4.4:80
```

HAProxy 역할:

```text
HTTP 80 수신
HTTPS 443 수신
Self-Signed SSL 인증서 사용
HTTP → HTTPS 리다이렉트
X-Forwarded-For 전달
X-Forwarded-Proto 전달
X-Forwarded-Port 전달
Web1 Swarm Published Port로 Reverse Proxy
```

HAProxy는 더 이상 여러 Web 노드에 대한 실제 로드밸런싱을 담당하지 않는다.

요청 분산은 Docker Swarm이 담당한다.

```text
HAProxy Reverse Proxy
→ Web1:80
→ Swarm Ingress Load Balancing
→ Replica 1 또는 Replica 2
```

`templates/haproxy.cfg.j2`와 `proxy.yml`은 아직 Swarm 단일 Backend 구조로 최종 수정하지 않았다.

# 19. 기존 환경 수동 정리 대상

이번 프로젝트는 전환 자동화가 아니라 처음부터 Swarm을 사용하는 Playbook 구조로 작성한다.

따라서 기존 Compose 환경의 일회성 정리 작업은 Playbook에 포함하지 않는다.

실제 Swarm Stack 배포 전에 수동으로 확인할 항목:

```text
기존 dandelion-wordpress 컨테이너
기존 Compose 프로젝트
기존 80번 포트 점유
기존 재시작 정책
기존 Stack 또는 Service 유무
```

예상 수동 정리 흐름:

```text
기존 WordPress 컨테이너 재시작 정책 해제
→ 기존 컨테이너 중지 및 제거
→ 기존 Compose 프로젝트 정리
→ 80번 포트 해제 확인
→ WordPress 데이터 볼륨은 유지
→ Swarm Stack 배포
```

볼륨 삭제 옵션인 `-v`는 사용하지 않는다.

유지 대상:

```text
dandelion_wordpress_data
```
