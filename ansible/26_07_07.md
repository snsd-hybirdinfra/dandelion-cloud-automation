# 2. WordPress Swarm Service 0/2 원인 조사

## 2.1 Task 상태 확인

WordPress Service 상태가 `0/2`로 표시되어 Task 이력과 로그를 확인하였다.

```bash
ansible web1-node -b -m shell -a "
docker service ps dandelion-wp_wordpress --no-trunc
"
```

과거 Task에서는 다음 오류가 확인되었다.

```text
task: non-zero exit (137): dockerexec: unhealthy container
```

그러나 최신 Task는 오류 종료가 아니라 다음 상태였다.

```text
Desired State: Shutdown
Current State: Complete
```

Apache 로그에서도 비정상 Crash가 아니라 정상 종료 신호가 확인되었다.

```text
AH00170: caught SIGWINCH, shutting down gracefully
```

## 2.2 Swarm 및 Service 상태 확인

다음 항목을 확인하였다.

```text
Web1 Node Status       Ready
Node Availability     Active
Manager Status        Leader
Service Desired       Replicas=2
Update Status         completed
```

따라서 Docker 자동 시작이나 Swarm Node 상태 문제는 아니었다.

## 2.3 재시작 정책 수정

상시 서비스인 WordPress가 정상 종료로 기록되더라도 다시 실행되도록 Stack Template의 재시작 조건을 수정하였다.

기존:

```yaml
restart_policy:
  condition: on-failure
  delay: 5s
```

수정:

```yaml
restart_policy:
  condition: any
  delay: 5s
```

수정 파일:

```text
templates/wordpress-stack.yml.j2
```

`condition: any`는 Task가 오류로 종료되거나 정상 종료되더라도 Swarm이 다시 생성하도록 한다.

> 최종 `2/2` 상태 확인 결과는 이번 작업 로그에 별도로 남지 않았으므로 이후 전체 검증 시 다시 확인해야 한다.

# 3. MariaDB 수동 이중화 구성 분석

팀원이 작성한 DB Replication 문서를 기준으로 수동 구성 내용을 분석하였다.

수동 구성 흐름:

```text
DB Primary
→ binary log 활성화
→ server-id=1
→ 복제 전용 사용자 생성

DB Replica
→ server-id=2
→ relay log 설정
→ CHANGE MASTER TO
→ START SLAVE

Proxy
→ HAProxy 3306 TCP Frontend
→ Primary 일반 서버
→ Replica backup 서버

Web
→ WORDPRESS_DB_HOST=192.168.4.32:3306
```

## 3.1 실제 DB 설정 확인

Primary:

```text
Hostname        db-primary
MariaDB         active / enabled
log_bin         mysql-bin
server-id       1
bind-address    0.0.0.0
```

Replica:

```text
Hostname        db-replica
MariaDB         active / enabled
log_bin         mysql-bin
server-id       2
relay_log       mysql-relay-bin
bind-address    0.0.0.0
```

## 3.2 복제 상태 확인

확인 당시 결과:

```text
Master_Host             192.168.4.195
Master_User             db_replica
Slave_IO_Running        Yes
Slave_SQL_Running       Yes
Seconds_Behind_Master   0
Last_IO_Error           없음
Last_SQL_Error          없음
```

Primary와 Replica의 Binary Log 파일 및 Position도 동일하였다.

## 3.3 HAProxy 및 WordPress DB 연결 확인

HAProxy:

```text
bind *:3306
Primary  192.168.4.195:3306
Replica  192.168.4.222:3306 backup
```

WordPress:

```text
WORDPRESS_DB_HOST=192.168.4.32:3306
```

최종 DB 접속 구조:

```text
WordPress
→ Proxy 192.168.4.32:3306
→ HAProxy TCP Proxy
→ DB Primary
→ 장애 시 DB Replica
```

# 5. MariaDB Replication 설정 템플릿 작성

다음 파일을 생성하였다.

```text
templates/mariadb/60-primary-replication.cnf.j2
templates/mariadb/60-replica-replication.cnf.j2
```

## 5.1 Primary Template

```ini
# Ansible managed - MariaDB Primary Replication

[mysqld]
server_id = {{ mariadb_primary_server_id }}
log_bin = {{ mariadb_binlog_name }}
binlog_format = ROW

bind-address = {{ mariadb_bind_address }}

expire_logs_days = 7
max_binlog_size = 100M
```

## 5.2 Replica Template

```ini
# Ansible managed - MariaDB Replica Replication

[mysqld]
server_id = {{ mariadb_replica_server_id }}

log_bin = {{ mariadb_binlog_name }}
relay_log = {{ mariadb_relay_log_name }}

binlog_format = ROW
log_slave_updates = ON
relay_log_recovery = ON

bind-address = {{ mariadb_bind_address }}
```

Replica의 `read_only=ON`은 이번 프로젝트의 HAProxy 장애 전환 테스트에서 Replica가 쓰기 요청을 받아야 하므로 적용하지 않았다.

# 6. 기존 단일 DB 자동화 확장

기존 `playbooks/database.yml`은 `hosts: db`만 대상으로 하여 Primary 한 대에만 MariaDB를 설치하였다.

이를 다음 구조로 변경하였다.

```text
첫 번째 Play
hosts: db:db_replica
→ Primary와 Replica 모두 MariaDB 설치
→ bind-address 설정
→ 서비스 활성화 및 시작
→ 익명 사용자 및 test DB 제거

두 번째 Play
hosts: db
→ Primary에만 wordpress_db 생성
→ Proxy IP용 wp_web 계정 생성
```

수정 파일:

```text
playbooks/database.yml
```

주요 변경점:

```yaml
hosts: "db:db_replica"
```

Primary 전용 WordPress DB 구성은 별도 Play로 분리하였다.

```yaml
hosts: db
```

WordPress가 HAProxy를 통해 DB에 접속하므로 DB 사용자 허용 Host도 Web IP가 아니라 Proxy IP를 기준으로 생성하도록 변경하였다.

# 7. database_replication.yml 1차 작성 및 방향 수정

처음에는 현재 수동으로 연결된 Replication을 유지·검증하는 Playbook을 작성하였다.

1차 기능:

```text
Primary/Replica 설정 파일 배포
Replication 사용자 상태 보장
기존 Primary 연결 정보 검증
중지된 Slave Thread 재시작
IO/SQL Thread 상태 검증
```

그러나 이 방식은 `SHOW SLAVE STATUS`가 비어 있는 신규 Replica에서는 중단되므로, 처음부터 자동 구축한다는 프로젝트 목표에 맞지 않았다.

따라서 다음 요구사항을 반영하여 전면 수정하였다.

```text
빈 DB Primary와 빈 DB Replica에서 시작
Primary DB Dump 자동 생성
Dump 시점 Binary Log 좌표 자동 추출
Dump를 Replica에 자동 Import
CHANGE MASTER TO 자동 실행
START SLAVE 자동 실행
최종 복제 상태 자동 검증
```

# 8. 신규 Replica 최초 동기화 자동화 구현

최종 `playbooks/database_replication.yml`은 Replica 상태에 따라 두 경로로 동작한다.

## 8.1 기존 복제가 있는 경우

```text
SHOW SLAVE STATUS 확인
→ Primary Host 및 Replication User 검증
→ Dump 및 Import 생략
→ 중지된 Thread가 있으면 START SLAVE
→ 최종 상태 검증
```

## 8.2 기존 복제가 없는 경우

```text
Primary WordPress DB Dump 생성
→ --single-transaction 사용
→ --master-data=2로 Binlog 좌표 기록
→ Dump에서 MASTER_LOG_FILE / MASTER_LOG_POS 추출
→ Control Node로 Fetch
→ Replica로 Copy
→ Replica에 Dump Import
→ Proxy용 WordPress DB 사용자 생성
→ CHANGE MASTER TO 실행
→ START SLAVE 실행
→ 임시 Dump 정리
→ IO/SQL Thread 및 DB 존재 여부 검증
```

주요 Dump 옵션:

```text
--single-transaction
--quick
--routines
--events
--triggers
--hex-blob
--master-data=2
--add-drop-database
--databases wordpress_db
```

임시 파일 흐름:

```text
Primary /tmp Dump
→ Control Node /tmp/dandelion-ansible-bootstrap
→ Replica /tmp Dump
→ Import 후 전체 삭제
```

# 10. database.yml Check Mode 문제 확인

`database.yml` 문법 검사는 성공하였다.

```bash
ansible-playbook --syntax-check playbooks/database.yml
```

그러나 `--check --diff` 실행에서는 조회 Task가 건너뛰어져 검증이 실패하였다.

증상:

```text
MariaDB 포트 응답 대기        skipped
MariaDB 버전 확인              skipped
Primary DB 조회                skipped
wordpress_database_result      빈 값
Primary WordPress DB 검증      실패
```

실제 DB가 없어서 실패한 것이 아니라 Check Mode에서 `command`와 일부 조회 Task가 실행되지 않아 등록 변수가 비어 있었던 문제였다.

수정 대상 Task:

```text
MariaDB 포트 응답 대기
MariaDB 버전 확인
Primary WordPress 데이터베이스 확인
```

추가할 설정:

```yaml
check_mode: false
```

수정 후 다시 수행해야 할 명령:

```bash
ansible-playbook playbooks/database.yml --check --diff
ansible-playbook playbooks/database.yml
ansible-playbook playbooks/database_replication.yml
```
