# 3. 기존 HAProxy 환경 확인

```text
active
enabled
```

## 3.2 기존 설정 문법 확인

```bash
ansible proxy-node -b -m command -a \
"haproxy -c -f /etc/haproxy/haproxy.cfg"
```

결과:

```text
Configuration file is valid
```

## 3.3 기존 수동 설정 내용

기존 `/etc/haproxy/haproxy.cfg`에는 다음 구성이 존재했다.

```text
HTTP 80 포트 수신
HTTPS 443 포트 수신
SSL 인증서 사용
HTTP → HTTPS 리다이렉트
X-Forwarded-Proto 헤더 설정
X-Forwarded-Port 헤더 설정
Web1·Web2 Round Robin 로드밸런싱
```

주요 기존 설정:

```haproxy
frontend proxy_front
    bind *:80
    bind *:443 ssl crt /etc/haproxy/certs/dandelion-wp.pem
    redirect scheme https if !{ ssl_fc }
    default_backend web_backend

backend web_backend
    balance roundrobin
    http-request set-header X-Forwarded-Proto https if { ssl_fc }
    http-request set-header X-Forwarded-Port 443 if { ssl_fc }
    server web1 192.168.4.4:80 check
    server web2 192.168.4.35:80 check
```

## 3.4 HAProxy 포트 확인

```bash
ansible proxy-node -b -m shell -a \
"ss -lntp '( sport = :80 )'"
```

결과:

```text
0.0.0.0:80 → haproxy
```

HAProxy가 80번 포트를 정상적으로 사용하고 있었다.

## 3.5 Proxy에서 Web 노드 직접 응답 확인

```bash
ansible proxy-node -m shell -a \
"for ip in 192.168.4.4 192.168.4.35; do echo WEB=\$ip; curl -sS -o /dev/null -w '%{http_code}\n' http://\$ip; done"
```

결과:

```text
192.168.4.4  → 302
192.168.4.35 → 302
```

두 WordPress Web 노드 모두 Proxy에서 정상 접근 가능했다.

# 4. 기존 설정과 Ansible 템플릿 비교

초기 `templates/haproxy.cfg.j2`에는 HTTP 80 포트와 Web Backend 로드밸런싱만 포함되어 있었다.

초기 템플릿에 없던 기존 기능:

```text
HTTPS 443 포트
SSL 인증서
HTTP → HTTPS 리다이렉트
X-Forwarded-Proto
X-Forwarded-Port
SSL 기본 암호화 설정
```

초기 템플릿을 그대로 적용하면 기존 HTTPS 구성이 사라질 수 있으므로 실제 실행을 중단하고 템플릿을 먼저 수정하였다.

# 5. HAProxy 변수 추가

`group_vars/all.yml`에 HTTPS 관련 변수를 추가하였다.

```yaml
haproxy_frontend_port: 80
haproxy_https_port: 443
haproxy_backend_port: 80
haproxy_balance_algorithm: roundrobin
haproxy_ssl_cert_path: "/etc/haproxy/certs/dandelion-wp.pem"
```

변수 의미:

| 변수 | 값 | 용도 |
|---|---:|---|
| `haproxy_frontend_port` | `80` | HTTP Frontend 포트 |
| `haproxy_https_port` | `443` | HTTPS Frontend 포트 |
| `haproxy_backend_port` | `80` | WordPress Web Backend 포트 |
| `haproxy_balance_algorithm` | `roundrobin` | 로드밸런싱 방식 |
| `haproxy_ssl_cert_path` | 인증서 경로 | HAProxy SSL 인증서 |

인증서 파일도 Proxy 노드에 존재하는지 확인하였다.

```text
/etc/haproxy/certs/dandelion-wp.pem
```

# 6. haproxy.cfg.j2 수정

수정한 템플릿:

```text
templates/haproxy.cfg.j2
```

최종 구조:

```haproxy
# Ansible에서 자동 생성한 HAProxy 설정 파일

global
    log /dev/log local0
    log /dev/log local1 notice
    chroot /var/lib/haproxy
    stats socket /run/haproxy/admin.sock mode 660 level admin
    stats timeout 30s
    user haproxy
    group haproxy
    daemon

    ca-base /etc/ssl/certs
    crt-base /etc/ssl/private

    ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384
    ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256
    ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets

defaults
    log global
    mode http
    option httplog
    option dontlognull
    option forwardfor
    option http-server-close

    timeout connect 5s
    timeout client 50s
    timeout server 50s
    timeout http-request 10s
    timeout http-keep-alive 10s

    errorfile 400 /etc/haproxy/errors/400.http
    errorfile 403 /etc/haproxy/errors/403.http
    errorfile 408 /etc/haproxy/errors/408.http
    errorfile 500 /etc/haproxy/errors/500.http
    errorfile 502 /etc/haproxy/errors/502.http
    errorfile 503 /etc/haproxy/errors/503.http
    errorfile 504 /etc/haproxy/errors/504.http

frontend wordpress_frontend
    bind *:{{ haproxy_frontend_port }}
    bind *:{{ haproxy_https_port }} ssl crt {{ haproxy_ssl_cert_path }}

    redirect scheme https code 301 if !{ ssl_fc }

    http-request set-header X-Forwarded-Proto https if { ssl_fc }
    http-request set-header X-Forwarded-Port {{ haproxy_https_port }} if { ssl_fc }

    default_backend wordpress_backend

backend wordpress_backend
    balance {{ haproxy_balance_algorithm }}
    option httpchk GET /

{% for host in groups['web'] %}
    server {{ host }} {{ hostvars[host].ansible_host }}:{{ haproxy_backend_port }} check inter 3s fall 3 rise 2
{% endfor %}
```

템플릿 적용 시 Inventory의 `web` 그룹을 기준으로 다음 Backend가 자동 생성된다.

```text
web1-node 192.168.4.4:80
web2-node 192.168.4.35:80
```

# 8. proxy.yml Check Mode 첫 번째 오류

실행:

```bash
ansible-playbook playbooks/proxy.yml --check --diff
```

첫 번째 오류:

```text
'dict object' has no attribute 'status'
```

오류가 발생한 조건:

```yaml
until: web_backend_check.status in [200, 301, 302]
```

## 8.1 원인

Check Mode에서 `ansible.builtin.uri`가 실제 HTTP 요청을 수행하지 않아 등록 변수에 `status`가 없었다.

## 8.2 수정

HTTP 상태 확인 Task에 다음 Task 옵션을 추가하였다.

```yaml
check_mode: false
changed_when: false
```

적용 대상:

```text
Proxy 노드에서 Web 노드 HTTP 응답 확인
HAProxy 경유 WordPress HTTP 응답 확인
```

`check_mode: false`는 전체 Playbook을 `--check`로 실행하더라도 해당 조회 Task는 실제로 수행하도록 한다.

# 9. proxy.yml Check Mode 두 번째 오류

두 번째 Check Mode에서는 마지막 Debug Task에서 오류가 발생하였다.

```text
list object has no element 0
```

문제가 된 값:

```yaml
haproxy_version_result.stdout_lines[0]
```

## 9.1 원인

Check Mode에서 다음 조회 Task가 생략되어 등록 변수에 실제 출력값이 없었다.

```text
HAProxy 포트 응답 대기
HAProxy 설정 문법 확인
HAProxy 버전 확인
```

## 9.2 수정

다음 조회 및 검증 Task에 `check_mode: false`를 추가하였다.

```yaml
- name: HAProxy 포트 응답 대기
  ansible.builtin.wait_for:
    host: 127.0.0.1
    port: "{{ haproxy_frontend_port }}"
    timeout: 60
  check_mode: false
  changed_when: false

- name: HAProxy 설정 문법 확인
  ansible.builtin.command:
    cmd: haproxy -c -f /etc/haproxy/haproxy.cfg
  register: haproxy_config_check
  changed_when: false
  check_mode: false

- name: HAProxy 버전 확인
  ansible.builtin.command:
    cmd: haproxy -v
  register: haproxy_version_result
  changed_when: false
  check_mode: false
```

# 10. check_mode 들여쓰기 오류

수정 과정에서 `check_mode: false`가 `wait_for` 모듈 내부에 들어가 다음 오류가 발생하였다.

```text
Unsupported parameters for (ansible.builtin.wait_for) module: check_mode
```

잘못된 형태:

```yaml
ansible.builtin.wait_for:
  host: 127.0.0.1
  port: 80
  timeout: 60
  check_mode: false
```

`check_mode`는 모듈 매개변수가 아니라 Task 옵션이므로 다음처럼 모듈과 같은 수준에 배치해야 한다.

```yaml
ansible.builtin.wait_for:
  host: 127.0.0.1
  port: 80
  timeout: 60
check_mode: false
changed_when: false
```

들여쓰기를 수정한 후 문법 검사와 Check Mode를 다시 실행하였다.
