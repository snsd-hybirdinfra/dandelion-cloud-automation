# Phase 1 Ansible 자동화 정리  
## Docker 설치부터 Custom WordPress 배포까지

## 1. 문서 목적

본 문서는 Team Dandelion 프로젝트에서 기존 `common.yml` 작성 이후 진행한 Ansible 자동화 작업을 정리한다.

기존 문서에 포함된 다음 내용은 제외한다.

```text
Ansible 설치
ansible.cfg 작성
inventory.ini 작성
Ansible Ping 및 sudo 검증
site.yml 작성
common.yml 작성 및 설명
common.yml 문법 검사
```

본 문서에는 다음 작업만 포함한다.

```text
docker.yml 작성
→ Docker Engine 및 Docker Compose Plugin 설치 자동화
→ requirements.yml 작성
→ group_vars/all.yml 작성
→ database.yml 작성
→ MariaDB 및 WordPress DB 구성 자동화
→ Custom WordPress 배포 파일 작성
→ wordpress.yml 작성
→ 각 Playbook 코드 구조 및 실행 흐름 설명
→ 문법 검사 명령 정리
```

현재 모든 Playbook은 **작성 및 문법 검사 단계**이며, 실제 노드 적용은 나중에 진행하기로 하였다.

## 2. 현재 서비스 설치 방식

팀 결정에 따른 서비스 배치 방식은 다음과 같다.

| 노드 그룹 | 서비스 | 설치 방식 |
|---|---|---|
| `web` | WordPress | Docker 컨테이너 |
| `db` | MariaDB | Ubuntu 패키지로 직접 설치 |
| `proxy` | HAProxy | Ubuntu 패키지로 직접 설치 |
| `backup` | 백업 및 검증 | 이후 Playbook에서 구성 |
| `monitoring` | 모니터링 | 이후 단계에서 구성 |

Docker는 WordPress 컨테이너를 실행하는 Web 노드에만 설치한다.

현재 Web 노드는 다음 두 대이다.

```ini
[web]
web1-node ansible_host=192.168.4.4
web2-node ansible_host=192.168.4.35
```

DB 노드는 다음과 같다.

```ini
[db]
db-node ansible_host=192.168.4.201
```

# 3. Docker 자동화 구성

## 3.1 docker.yml 목적

`docker.yml`은 Inventory의 `web` 그룹을 대상으로 다음 작업을 자동화한다.

```text
Docker 저장소 준비 패키지 설치
→ 충돌 가능 패키지 제거
→ Docker 공식 GPG 키 등록
→ Docker 공식 APT 저장소 등록
→ Docker Engine 설치
→ Docker Compose Plugin 설치
→ Docker 서비스 시작 및 부팅 시 자동 실행
→ ubuntu 사용자를 docker 그룹에 추가
→ Docker 및 Compose 동작 확인
```

실행 대상은 다음과 같다.

```yaml
hosts: web
```

따라서 `web1-node`, `web2-node`에만 적용된다.

---

## 3.2 docker.yml 작성

파일 위치:

```text
~/dandeliondir/ansible/playbooks/docker.yml
```

작성 내용:

```yaml
---
- name: Web 노드 Docker 환경 구성
  hosts: web
  become: true
  gather_facts: true

  vars:
    docker_arch_map:
      x86_64: amd64
      aarch64: arm64
      armv7l: armhf
      ppc64le: ppc64el
      s390x: s390x

    docker_apt_arch: >-
      {{ docker_arch_map.get(ansible_architecture, ansible_architecture) }}

  tasks:
    - name: Docker 저장소 등록에 필요한 패키지 설치
      ansible.builtin.apt:
        name:
          - ca-certificates
          - curl
        state: present
        update_cache: true
        cache_valid_time: 3600

    - name: Docker와 충돌할 수 있는 기존 패키지 제거
      ansible.builtin.apt:
        name:
          - docker.io
          - docker-compose
          - docker-compose-v2
          - docker-doc
          - podman-docker
          - containerd
          - runc
        state: absent

    - name: APT 키 저장 디렉터리 생성
      ansible.builtin.file:
        path: /etc/apt/keyrings
        state: directory
        owner: root
        group: root
        mode: "0755"

    - name: Docker 공식 GPG 키 다운로드
      ansible.builtin.get_url:
        url: https://download.docker.com/linux/ubuntu/gpg
        dest: /etc/apt/keyrings/docker.asc
        owner: root
        group: root
        mode: "0644"
        force: false

    - name: Docker 공식 APT 저장소 등록
      ansible.builtin.copy:
        dest: /etc/apt/sources.list.d/docker.sources
        owner: root
        group: root
        mode: "0644"
        content: |
          Types: deb
          URIs: https://download.docker.com/linux/ubuntu
          Suites: {{ ansible_distribution_release }}
          Components: stable
          Architectures: {{ docker_apt_arch }}
          Signed-By: /etc/apt/keyrings/docker.asc

    - name: Docker 패키지 설치
      ansible.builtin.apt:
        name:
          - docker-ce
          - docker-ce-cli
          - containerd.io
          - docker-buildx-plugin
          - docker-compose-plugin
        state: present
        update_cache: true

    - name: Docker 서비스 활성화 및 시작
      ansible.builtin.systemd_service:
        name: docker
        state: started
        enabled: true

    - name: ubuntu 사용자를 docker 그룹에 추가
      ansible.builtin.user:
        name: ubuntu
        groups:
          - docker
        append: true

    - name: SSH 연결 재설정
      ansible.builtin.meta: reset_connection

    - name: Docker 버전 확인
      ansible.builtin.command:
        cmd: docker --version
      register: docker_version_result
      changed_when: false
      become: false

    - name: Docker Compose 버전 확인
      ansible.builtin.command:
        cmd: docker compose version
      register: docker_compose_version_result
      changed_when: false
      become: false

    - name: ubuntu 사용자의 Docker 데몬 접근 확인
      ansible.builtin.command:
        cmd: docker ps
      changed_when: false
      become: false

    - name: Docker 설치 결과 출력
      ansible.builtin.debug:
        msg:
          - "{{ docker_version_result.stdout }}"
          - "{{ docker_compose_version_result.stdout }}"
```

## 3.3 docker.yml 주요 코드 설명

### CPU 아키텍처 변환

Ansible은 일반적으로 64비트 Intel/AMD CPU를 `x86_64`로 수집하지만 Docker APT 저장소에서는 `amd64`라는 이름을 사용한다.

```yaml
docker_arch_map:
  x86_64: amd64
```

다음 변수는 현재 노드의 CPU 아키텍처를 Docker 저장소 형식으로 변환한다.

```yaml
docker_apt_arch: >-
  {{ docker_arch_map.get(ansible_architecture, ansible_architecture) }}
```

### 충돌 패키지 제거

```yaml
state: absent
```

Ubuntu 기본 저장소의 `docker.io`, 기존 `containerd`, `runc` 등이 Docker 공식 패키지와 충돌하는 것을 방지하기 위해 제거한다.

이미 해당 패키지가 없다면 변경 없이 넘어간다.

### Docker 공식 저장소 등록

다음 파일을 생성한다.

```text
/etc/apt/sources.list.d/docker.sources
```

저장소 내용에는 Ubuntu 코드명과 CPU 아키텍처가 자동으로 반영된다.

```yaml
Suites: {{ ansible_distribution_release }}
Architectures: {{ docker_apt_arch }}
```

Ubuntu 24.04 환경이라면 일반적으로 `ansible_distribution_release`는 `noble`이 된다.

### Docker 패키지 설치

```yaml
- docker-ce
- docker-ce-cli
- containerd.io
- docker-buildx-plugin
- docker-compose-plugin
```

구형 `docker-compose` 명령이 아니라 다음 형식의 Compose V2 명령을 사용한다.

```bash
docker compose version
docker compose up -d
```

### Docker 서비스 관리

```yaml
state: started
enabled: true
```

Docker 서비스를 현재 시작하고 서버 재부팅 후에도 자동 실행되게 한다.

### docker 그룹 추가

```yaml
append: true
```

`ubuntu` 사용자의 기존 보조 그룹은 유지하면서 `docker` 그룹만 추가한다.

그룹 변경 사항을 새 SSH 세션에 반영하기 위해 다음 Task를 실행한다.

```yaml
ansible.builtin.meta: reset_connection
```

### 동작 확인

다음 명령을 Web 노드에서 실행한다.

```bash
docker --version
docker compose version
docker ps
```

확인 작업은 서버 상태를 변경하지 않으므로 다음 설정을 사용한다.

```yaml
changed_when: false
```

# 4. Ansible Collection 구성

## 4.1 requirements.yml 목적

기본 Ansible 모듈 외에 다음 Collection을 사용한다.

| Collection | 용도 |
|---|---|
| `ansible.mysql` | MariaDB 데이터베이스 및 사용자 관리 |
| `community.docker` | Docker Compose V2 프로젝트 관리 |

파일 위치:

```text
~/dandeliondir/ansible/requirements.yml
```

작성 내용:

```yaml
---
collections:
  - name: ansible.mysql
  - name: community.docker
```

Collection 설치 명령:

```bash
cd ~/dandeliondir/ansible
ansible-galaxy collection install -r requirements.yml
```

설치된 Collection 확인:

```bash
ansible-galaxy collection list
```

이 명령은 관리 노드에 MariaDB나 Docker를 설치하는 작업이 아니다. Control Node에 Ansible 모듈을 설치하는 작업이다.

# 5. 공통 변수 구성

## 5.1 group_vars/all.yml 목적

`database.yml`과 `wordpress.yml`에서 동일한 DB 정보와 WordPress 설정을 사용하기 위해 공통 변수 파일을 작성한다.

파일 위치:

```text
~/dandeliondir/ansible/group_vars/all.yml
```

작성 내용:

```yaml
---
# WordPress 데이터베이스 설정
wordpress_db_name: wordpress
wordpress_db_user: wordpress
wordpress_db_password: "CHANGE_ME_STRONG_PASSWORD"

# MariaDB 네트워크 설정
mariadb_bind_address: "0.0.0.0"
mariadb_port: 3306

# Custom WordPress 이미지 설정
wordpress_base_image: "wordpress:7.0.0-php8.4-apache"
wordpress_image_name: "dandelion-wordpress"
wordpress_image_tag: "v1"

# WordPress 컨테이너 설정
wordpress_container_name: "dandelion-wordpress"
wordpress_host_port: 80
wordpress_data_volume: "dandelion_wordpress_data"
```

### 주의사항

`wordpress_db_password`는 실제 실행 전에 안전한 비밀번호로 변경해야 한다.

```yaml
wordpress_db_password: "CHANGE_ME_STRONG_PASSWORD"
```

`database.yml`에서 생성하는 DB 사용자 비밀번호와 `wordpress.yml`에서 WordPress 컨테이너에 전달하는 비밀번호는 반드시 같아야 한다.

GitHub에 올리기 전에는 비밀번호를 Ansible Vault로 암호화하는 것이 좋다.

또한 `wordpress_base_image`에 지정한 태그는 실제 실행 전에 Docker Hub에서 존재 여부를 확인해야 한다. 사용할 태그가 확정되지 않았다면 팀에서 정한 공식 WordPress 이미지 태그로 변경한다.

# 6. MariaDB 자동화 구성

## 6.1 database.yml 목적

`database.yml`은 Inventory의 `db` 그룹을 대상으로 다음 작업을 수행한다.

```text
MariaDB 서버 및 클라이언트 설치
→ PyMySQL 설치
→ 외부 접속을 위한 bind-address 변경
→ MariaDB 서비스 시작
→ 익명 사용자 제거
→ test 데이터베이스 제거
→ WordPress 데이터베이스 생성
→ Web 노드 IP별 WordPress DB 계정 생성
→ MariaDB 버전 확인
```

실행 대상:

```yaml
hosts: db
```

현재 대상은 `db-node` 한 대이다.

---

## 6.2 database.yml 작성

파일 위치:

```text
~/dandeliondir/ansible/playbooks/database.yml
```

작성 내용:

```yaml
---
- name: DB 노드 MariaDB 환경 구성
  hosts: db
  become: true
  gather_facts: true

  tasks:
    - name: MariaDB 및 Python 연동 패키지 설치
      ansible.builtin.apt:
        name:
          - mariadb-server
          - mariadb-client
          - python3-pymysql
        state: present
        update_cache: true
        cache_valid_time: 3600

    - name: MariaDB 외부 접속 주소 설정
      ansible.builtin.lineinfile:
        path: /etc/mysql/mariadb.conf.d/50-server.cnf
        regexp: '^\s*bind-address\s*='
        line: "bind-address = {{ mariadb_bind_address }}"
        backup: true
      notify: MariaDB 서비스 재시작

    - name: MariaDB 서비스 활성화 및 시작
      ansible.builtin.systemd_service:
        name: mariadb
        state: started
        enabled: true

    - name: MariaDB 설정 변경 즉시 반영
      ansible.builtin.meta: flush_handlers

    - name: MariaDB 포트 응답 대기
      ansible.builtin.wait_for:
        host: 127.0.0.1
        port: "{{ mariadb_port }}"
        timeout: 30

    - name: MariaDB 익명 사용자 제거
      ansible.mysql.mysql_user:
        name: ""
        host_all: true
        state: absent
        login_unix_socket: /run/mysqld/mysqld.sock

    - name: MariaDB 테스트 데이터베이스 제거
      ansible.mysql.mysql_db:
        name: test
        state: absent
        login_unix_socket: /run/mysqld/mysqld.sock

    - name: WordPress 데이터베이스 생성
      ansible.mysql.mysql_db:
        name: "{{ wordpress_db_name }}"
        state: present
        encoding: utf8mb4
        collation: utf8mb4_unicode_ci
        login_unix_socket: /run/mysqld/mysqld.sock

    - name: Web 노드별 WordPress DB 사용자 생성
      ansible.mysql.mysql_user:
        name: "{{ wordpress_db_user }}"
        password: "{{ wordpress_db_password }}"
        host: "{{ hostvars[item].ansible_host }}"
        priv: "{{ wordpress_db_name }}.*:ALL"
        state: present
        login_unix_socket: /run/mysqld/mysqld.sock
      loop: "{{ groups['web'] }}"
      no_log: true

    - name: MariaDB 버전 확인
      ansible.builtin.command:
        cmd: mariadb --version
      register: mariadb_version_result
      changed_when: false

    - name: MariaDB 설치 결과 출력
      ansible.builtin.debug:
        msg: "{{ mariadb_version_result.stdout }}"

  handlers:
    - name: MariaDB 서비스 재시작
      ansible.builtin.systemd_service:
        name: mariadb
        state: restarted
```

## 6.3 database.yml 주요 코드 설명

### MariaDB 패키지 설치

```yaml
- mariadb-server
- mariadb-client
- python3-pymysql
```

`mariadb-server`는 DB 서버, `mariadb-client`는 접속 및 백업 명령, `python3-pymysql`은 Ansible MySQL 모듈이 MariaDB와 통신할 때 사용한다.

### bind-address 설정

기본값이 `127.0.0.1`이면 DB 노드 내부에서만 접속할 수 있다.

다음 변수값을 적용한다.

```yaml
mariadb_bind_address: "0.0.0.0"
```

결과:

```ini
bind-address = 0.0.0.0
```

이 설정으로 Web 노드가 MariaDB에 연결할 수 있다.

단, OpenStack 보안그룹에서는 DB 노드의 TCP 3306 포트를 인터넷 전체에 열지 말고 Web 노드에서만 접근하도록 제한해야 한다.

허용 대상:

```text
192.168.4.4
192.168.4.35
```

### Handler와 flush_handlers

설정 파일이 실제로 변경된 경우에만 Handler가 예약된다.

```yaml
notify: MariaDB 서비스 재시작
```

일반적으로 Handler는 Play 마지막에 실행되지만, 이후 DB 작업 전에 변경된 설정을 반영해야 하므로 다음 Task로 즉시 실행한다.

```yaml
ansible.builtin.meta: flush_handlers
```

### MariaDB 준비 대기

```yaml
host: 127.0.0.1
port: 3306
timeout: 30
```

서비스 시작 직후 DB가 준비되기 전에 사용자 생성 Task가 실행되는 것을 방지한다.

### 익명 사용자 제거

```yaml
name: ""
host_all: true
state: absent
```

MariaDB의 이름 없는 익명 계정을 모든 Host 범위에서 제거한다.

### 테스트 DB 제거

```yaml
name: test
state: absent
```

기본 테스트 데이터베이스가 있으면 제거한다.

### WordPress DB 생성

```yaml
name: "{{ wordpress_db_name }}"
encoding: utf8mb4
collation: utf8mb4_unicode_ci
```

현재 변수 기준으로 `wordpress` 데이터베이스가 생성된다.

`state: present`이므로 이미 존재하면 다시 생성하지 않는다.

### Web 노드별 DB 사용자 생성

```yaml
loop: "{{ groups['web'] }}"
```

Inventory의 `web` 그룹을 순회한다.

```yaml
host: "{{ hostvars[item].ansible_host }}"
```

각 Web 노드의 실제 IP를 가져온다.

현재 생성 대상은 다음과 같다.

```text
wordpress@192.168.4.4
wordpress@192.168.4.35
```

즉 동일한 `wordpress` 사용자 이름이라도 접속 원본 IP별로 별도 계정을 생성한다.

권한 범위:

```yaml
priv: "{{ wordpress_db_name }}.*:ALL"
```

현재 값으로 변환하면 다음과 같다.

```text
wordpress.*:ALL
```

WordPress DB에 대해서만 권한을 부여한다.

### Unix Socket 인증

```yaml
login_unix_socket: /run/mysqld/mysqld.sock
```

DB 노드 내부에서 root 비밀번호 대신 Ubuntu/MariaDB의 Unix Socket 인증으로 관리자 작업을 수행한다.

수동 명령의 다음 방식과 유사하다.

```bash
sudo mariadb
```

### 로그 보호


# 8. Docker Compose 템플릿 구성

## 8.1 wordpress-compose.yml.j2 작성

파일 위치:

```text
~/dandeliondir/ansible/templates/wordpress-compose.yml.j2
```

작성 내용:

```yaml
---
services:
  wordpress:
    build:
      context: .
      args:
        WORDPRESS_BASE_IMAGE: "{{ wordpress_base_image }}"

    image: "{{ wordpress_image_name }}:{{ wordpress_image_tag }}"
    container_name: "{{ wordpress_container_name }}"
    restart: unless-stopped

    ports:
      - "{{ wordpress_host_port }}:80"

    environment:
      WORDPRESS_DB_HOST: "{{ hostvars[groups['db'][0]].ansible_host }}:{{ mariadb_port }}"
      WORDPRESS_DB_NAME: "{{ wordpress_db_name }}"
      WORDPRESS_DB_USER: "{{ wordpress_db_user }}"
      WORDPRESS_DB_PASSWORD: "{{ wordpress_db_password }}"

    volumes:
      - wordpress_data:/var/www/html

    healthcheck:
      test:
        - CMD-SHELL
        - curl -fsS http://127.0.0.1/ || exit 1
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 60s

volumes:
  wordpress_data:
    name: "{{ wordpress_data_volume }}"
```

---

## 8.2 Compose 템플릿 설명

### Custom Image 빌드

```yaml
build:
  context: .
  args:
    WORDPRESS_BASE_IMAGE: "{{ wordpress_base_image }}"
```

Web 노드의 `/opt/dandelion/wordpress` 디렉터리를 빌드 컨텍스트로 사용한다.

Dockerfile의 다음 ARG에 변수값을 전달한다.

```dockerfile
ARG WORDPRESS_BASE_IMAGE
```

### 이미지 및 컨테이너 이름

```yaml
image: "{{ wordpress_image_name }}:{{ wordpress_image_tag }}"
container_name: "{{ wordpress_container_name }}"
```

현재 변수 기준:

```text
이미지: dandelion-wordpress:v1
컨테이너: dandelion-wordpress
```

각 Web 노드는 서로 다른 Docker Host이므로 동일한 컨테이너 이름을 사용해도 충돌하지 않는다.

### 포트 연결

```yaml
ports:
  - "{{ wordpress_host_port }}:80"
```

현재 변수값은 `80`이므로 다음과 같이 연결된다.

```text
Web 노드의 80번 포트 → WordPress 컨테이너의 80번 포트
```

### DB 연결 환경변수

```yaml
WORDPRESS_DB_HOST: "{{ hostvars[groups['db'][0]].ansible_host }}:{{ mariadb_port }}"
```

Inventory의 DB 그룹 첫 번째 노드 IP와 MariaDB 포트를 조합한다.

현재 값:

```text
192.168.4.201:3306
```

나머지 환경변수에는 공통 변수 파일의 DB 이름, 사용자, 비밀번호가 적용된다.

### WordPress 데이터 볼륨

```yaml
volumes:
  - wordpress_data:/var/www/html
```

WordPress 파일을 Docker 이름 있는 볼륨에 저장한다.

컨테이너가 다시 생성돼도 WordPress 파일을 유지하기 위한 구성이다.

현재 볼륨 이름:

```text
dandelion_wordpress_data
```

### 재시작 정책

```yaml
restart: unless-stopped
```

사용자가 명시적으로 중지한 경우를 제외하고, 컨테이너 종료 또는 서버 재부팅 후 다시 시작한다.

### Health Check

```yaml
curl -fsS http://127.0.0.1/ || exit 1
```

컨테이너 내부에서 Apache/WordPress의 HTTP 응답을 확인한다.

```text
30초 간격
10초 제한 시간
5회 재시도
시작 후 60초 준비 시간
```

# 9. WordPress 배포 자동화

## 9.1 wordpress.yml 목적

`wordpress.yml`은 Web 노드 두 대에 다음 작업을 수행한다.

```text
배포 디렉터리 생성
→ Dockerfile 복사
→ custom.ini 복사
→ 변수 기반 compose.yml 생성
→ 변경된 경우 Custom 이미지 재빌드
→ WordPress Compose 서비스 실행
→ Web 노드 80번 포트 확인
→ WordPress HTTP 응답 확인
→ 배포 결과 출력
```

실행 대상:

```yaml
hosts: web
```

---

## 9.2 wordpress.yml 작성

파일 위치:

```text
~/dandeliondir/ansible/playbooks/wordpress.yml
```

작성 내용:

```yaml
---
- name: Web 노드 Custom WordPress 배포
  hosts: web
  become: true
  gather_facts: true

  vars:
    wordpress_remote_dir: /opt/dandelion/wordpress

  tasks:
    - name: WordPress 배포 디렉터리 생성
      ansible.builtin.file:
        path: "{{ wordpress_remote_dir }}"
        state: directory
        owner: root
        group: root
        mode: "0755"

    - name: WordPress Dockerfile 복사
      ansible.builtin.copy:
        src: ../files/wordpress/Dockerfile
        dest: "{{ wordpress_remote_dir }}/Dockerfile"
        owner: root
        group: root
        mode: "0644"
      notify: Custom WordPress 이미지 재빌드 및 배포

    - name: WordPress PHP 설정 파일 복사
      ansible.builtin.copy:
        src: ../files/wordpress/custom.ini
        dest: "{{ wordpress_remote_dir }}/custom.ini"
        owner: root
        group: root
        mode: "0644"
      notify: Custom WordPress 이미지 재빌드 및 배포

    - name: WordPress Docker Compose 파일 생성
      ansible.builtin.template:
        src: ../templates/wordpress-compose.yml.j2
        dest: "{{ wordpress_remote_dir }}/compose.yml"
        owner: root
        group: root
        mode: "0600"
      notify: Custom WordPress 이미지 재빌드 및 배포
      no_log: true

    - name: 변경된 WordPress 배포 설정 즉시 반영
      ansible.builtin.meta: flush_handlers

    - name: WordPress Compose 서비스 실행 상태 보장
      community.docker.docker_compose_v2:
        project_src: "{{ wordpress_remote_dir }}"
        files:
          - compose.yml
        state: present
        build: policy
        remove_orphans: true

    - name: WordPress 서비스 포트 응답 대기
      ansible.builtin.wait_for:
        host: 127.0.0.1
        port: "{{ wordpress_host_port }}"
        timeout: 120

    - name: WordPress HTTP 응답 확인
      ansible.builtin.uri:
        url: "http://127.0.0.1:{{ wordpress_host_port }}"
        status_code:
          - 200
          - 301
          - 302
        return_content: false
      register: wordpress_http_result
      retries: 10
      delay: 5
      until: wordpress_http_result.status in [200, 301, 302]

    - name: WordPress 배포 결과 출력
      ansible.builtin.debug:
        msg:
          - "노드: {{ inventory_hostname }}"
          - "접속 주소: http://{{ ansible_host }}:{{ wordpress_host_port }}"
          - "HTTP 상태 코드: {{ wordpress_http_result.status }}"

  handlers:
    - name: Custom WordPress 이미지 재빌드 및 배포
      community.docker.docker_compose_v2:
        project_src: "{{ wordpress_remote_dir }}"
        files:
          - compose.yml
        state: present
        build: always
        remove_orphans: true
```

---

## 9.3 wordpress.yml 주요 코드 설명

### 실행 대상과 권한

```yaml
hosts: web
become: true
```

`web1-node`, `web2-node`에서 root 권한으로 실행한다.

### 배포 경로

```yaml
wordpress_remote_dir: /opt/dandelion/wordpress
```

각 Web 노드에는 다음 파일이 배치된다.

```text
/opt/dandelion/wordpress/
├── Dockerfile
├── custom.ini
└── compose.yml
```

### 디렉터리 생성

```yaml
state: directory
mode: "0755"
```

디렉터리가 없으면 생성하고, 이미 있으면 유지한다.

### 파일 복사

Dockerfile과 `custom.ini`를 Control Node에서 Web 노드로 복사한다.

```yaml
ansible.builtin.copy
```

파일 내용이 변경된 경우 다음 Handler가 예약된다.

```yaml
notify: Custom WordPress 이미지 재빌드 및 배포
```

### Compose 템플릿 생성

```yaml
ansible.builtin.template
```

Jinja2 템플릿의 `{{ 변수 }}`를 실제 값으로 변경해 Web 노드에 `compose.yml`을 생성한다.

DB 비밀번호가 포함되므로 권한을 다음과 같이 설정한다.

```yaml
mode: "0600"
```

root만 읽고 쓸 수 있다.

```yaml
no_log: true
```

비밀번호가 실행 로그에 표시되는 것을 막는다.

### Handler 즉시 실행

```yaml
ansible.builtin.meta: flush_handlers
```

Dockerfile, PHP 설정, Compose 파일 중 하나가 변경된 경우 이미지 빌드와 배포 Handler를 이 시점에 실행한다.

### Compose 서비스 상태 보장

```yaml
community.docker.docker_compose_v2
```

수동 명령의 다음 작업과 비슷하다.

```bash
cd /opt/dandelion/wordpress
docker compose -f compose.yml up -d
```

주요 옵션:

```yaml
state: present
```

컨테이너가 실행 중인 상태를 보장한다.

```yaml
build: policy
```

일반 Compose 빌드 정책을 따른다.

```yaml
remove_orphans: true
```

현재 Compose 파일에서 삭제된 이전 서비스 컨테이너를 정리한다.

### Handler의 강제 재빌드

```yaml
build: always
```

Dockerfile, `custom.ini`, `compose.yml`이 변경됐을 때 기존 이미지가 있어도 다시 빌드한다.

수동 명령과 유사한 형태:

```bash
docker compose -f compose.yml up -d --build
```

### 포트 확인

```yaml
host: 127.0.0.1
port: 80
timeout: 120
```

컨테이너 실행 후 Web 노드의 80번 포트가 열릴 때까지 최대 120초 기다린다.

### HTTP 응답 확인

```yaml
url: "http://127.0.0.1:{{ wordpress_host_port }}"
```

Web 노드 내부에서 WordPress에 접속한다.

성공으로 허용하는 상태 코드:

```text
200
301
302
```

WordPress 초기 설정이나 URL 처리 과정에서 리다이렉트가 발생할 수 있으므로 `301`, `302`도 허용한다.

재시도 설정:

```yaml
retries: 10
delay: 5
```

5초 간격으로 최대 10회 확인한다.

### 결과 출력

각 Web 노드의 이름, 내부 접속 주소, HTTP 상태 코드를 출력한다.

예시:

```text
노드: web1-node
접속 주소: http://192.168.4.4:80
HTTP 상태 코드: 200
```

내부 IP는 외부 사용자 PC에서 직접 접근하는 주소가 아닐 수 있다. 최종 외부 접속은 이후 HAProxy를 구성한 뒤 Proxy 노드 주소로 확인한다.

한 번에 전체 실행하기 전 개별 Playbook과 단일 노드 시험 실행을 권장한다.


## 13.2 Docker 전체 Web 노드 적용


## 13.3 DB 노드 적용


## 13.4 WordPress 단일 노드 시험
