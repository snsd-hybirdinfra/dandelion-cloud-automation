## 3. 수행 작업

### 3.1 서비스 구조 변경 검토

기존 서비스 대상이었던 Nginx 단일 컨테이너 구조를 재검토하였다.  
Nginx 단일 컨테이너는 구현 난이도는 낮지만, 백업/복구 검증의 의미가 약하고 Dockerfile 기반 커스텀 이미지 작성 근거가 부족하다고 판단하였다.

이에 따라 서비스 대상을 다음과 같이 변경하였다.

| 기존 | 변경 |
|---|---|
| Docker 기반 Nginx 컨테이너 | Custom WordPress + MariaDB |
| 정적 웹서비스 중심 | DB 연동 웹서비스 중심 |
| 단순 HTTP 응답 확인 | HTTP 확인 + DB dump + 파일 백업 |
| 백업 의미 약함 | DB 및 WordPress files 백업 가능 |

### 3.2 Custom WordPress Image 방향 정리

Docker Hub 공식 WordPress 이미지를 Base Image로 사용하고, Dockerfile을 통해 운영 설정 수준의 커스터마이징을 적용하는 방향으로 정리하였다.

생성 예정 파일은 다음과 같다.

| 경로 | 내용 |
|---|---|
| docker/wordpress/Dockerfile | WordPress 공식 이미지 기반 커스텀 이미지 정의 |
| docker/wordpress/custom.ini | PHP upload size, memory limit 등 운영 설정 |
| docker/wordpress/README.md | 커스텀 이미지 목적과 범위 설명 |
| docker/compose/docker-compose.yml | WordPress + MariaDB 서비스 구성 |

커스터마이징은 WordPress 기능 개발이 아니라 운영 설정 조정으로 제한하였다.

### 3.3 DB 볼륨 및 백업 구조 정리

MariaDB 원본 데이터는 Web Node의 Docker Volume에 저장하고, Backup / Validation Node에는 백업 파일만 저장하는 구조로 정리하였다.

정리된 기준은 다음과 같다.

| 대상 | 위치 |
|---|---|
| WordPress files volume | Web Node |
| MariaDB data volume | Web Node |
| MariaDB dump backup | Backup / Validation Node |
| WordPress files backup | Backup / Validation Node |
| docker-compose.yml backup | Backup / Validation Node |

DB 원본 볼륨을 원격 Backup Node에 직접 두는 방식은 기본 프로젝트 범위에서 제외하였다.

---

### 3.4 HAProxy Reverse Proxy 확장 검토

강사 피드백에 따라 LB 구성을 검토하였다.  
다만 OpenStack Octavia, LBaaS, 멀티 웹노드 HA는 범위가 크기 때문에 제외하였다.

대신 HAProxy Container를 사용하여 WordPress 서비스 앞단에 Reverse Proxy 계층을 두는 방향을 B급 선택 확장으로 정리하였다.

구성 방향은 다음과 같다.

~~~text
Client
→ HAProxy Reverse Proxy / LB Node
→ Web Node WordPress HTTP 80
→ MariaDB Container
~~~

HAProxy는 HTTPS Termination, HTTP to HTTPS Redirect, Backend 전달 역할을 수행한다.

### 3.5 모니터링 도구 범위 분류

모니터링 도구는 A급 필수 구현이 아니라 B급 선택 확장으로 분류하였다.

| 도구 | 분류 | 역할 |
|---|---|---|
| health_check.sh | A급 | 기본 상태 점검 |
| Prometheus | B급 | 메트릭 수집 |
| Grafana | B급 | 대시보드 시각화 |
| node_exporter | B급 | 서버 OS 메트릭 수집 |
| cAdvisor | B급 | 컨테이너 메트릭 수집 |

A급 구현 완료 전에는 모니터링 확장을 진행하지 않는 기준으로 정리하였다.

## 9. 추가 작업: 접속 방식 및 OpenStack CLI 구축 목표 반영

### 9.1 포트포워딩 기반 접속 방식 정리

실습 환경에서 외부 접속을 검증하기 위해 공유기 포트포워딩 기반 접속 방식을 우선 적용하기로 정리하였다.

정리된 접속 대상은 다음과 같다.

| 접속 유형 | 대상 | 목적 |
|---|---|---|
| SSH | 각 노드 | 서버 접속 및 Ansible 관리 확인 |
| HTTP | WordPress Service | 웹서비스 접속 확인 |
| HTTPS | HAProxy Reverse Proxy | B급 확장 시 HTTPS 접속 확인 |
| Grafana | Monitoring Node | B급 확장 시 대시보드 확인 |

포트포워딩은 실습 환경의 접근성을 확보하기 위한 방식이며, 문서에는 보안그룹 및 허용 포트 기준과 함께 기록하기로 하였다.

### 9.2 OpenStack CLI 기반 노드 구축 목표 정리

최종적으로는 OpenStack Horizon 대시보드에서 수동으로 노드를 생성하는 방식에 머무르지 않고, OpenStack CLI 명령어를 통해 노드 생성과 네트워크 구성을 수행하는 것을 목표로 정리하였다.

구현 목표는 다음과 같다.

| 구분 | 명령어 기반 구현 대상 |
|---|---|
| Network | openstack network create |
| Subnet | openstack subnet create |
| Router | openstack router create, openstack router add subnet |
| Security Group | openstack security group create, openstack security group rule create |
| Instance | openstack server create |
| Floating IP | openstack floating ip create, openstack server add floating ip |

이 방향을 통해 OpenStack 인프라 구성과 Ansible 서버 자동화를 하나의 명령어 기반 운영 흐름으로 연결할 수 있다.

### 9.3 작업 기준 변경

기존 작업 기준은 OpenStack 인프라가 준비된 상태에서 Ansible 자동화를 수행하는 방향에 가까웠다.  
금일 추가 결정에 따라 인프라 준비 단계도 OpenStack CLI 기반으로 문서화하고, 가능하면 실제 명령어 실행 결과를 캡처하기로 하였다.

변경된 기준은 다음과 같다.

| 기존 기준 | 변경 기준 |
|---|---|
| OpenStack 대시보드 기반 수동 구성 가능 | OpenStack CLI 기반 구성 명령어 정리 |
| 노드 준비 후 Ansible 실행 | OpenStack CLI로 노드 생성 후 Ansible 실행 |
| 접속 방식 미정 | 공유기 포트포워딩 기반 접속 검증 |
| 서비스 배포 중심 | 인프라 생성 → 접속 → 자동화 → 서비스 배포 → 검증 흐름 |

## 10. 추가 작업: B+ Stretch Goal 자동화 방향 정리

### 10.1 OpenStack CLI + Ansible 통합 자동화 방향

B급 구현이 조기에 완료될 경우, 추가 확장 목표로 OpenStack CLI 명령을 Ansible Playbook에 통합하는 방향을 정리하였다.

기존 구조는 OpenStack에서 노드가 준비된 이후 Ansible로 서버 설정과 서비스 배포를 수행하는 방식이다.  
추가 확장 구조에서는 Ansible이 OpenStack CLI 명령을 실행하여 인프라 생성 단계까지 자동화한다.

~~~text
기존:
OpenStack 노드 준비
→ Ansible 서버 설정
→ WordPress/MariaDB 배포
→ 검증

확장:
Ansible 실행
→ OpenStack CLI로 노드 생성
→ Inventory 반영
→ 서버 설정
→ WordPress/MariaDB 배포
→ HAProxy / Monitoring 구성
→ 검증
~~~

### 10.2 자동화 대상 정리

| 단계 | 자동화 대상 |
|---|---|
| 1 | OpenStack 인증 정보 확인 |
| 2 | Network / Subnet / Router 생성 |
| 3 | Security Group 및 포트 정책 생성 |
| 4 | Control / Web / Backup / Proxy Node 생성 |
| 5 | Floating IP 또는 포트포워딩 접속 정보 확인 |
| 6 | 생성된 노드 SSH 접속 확인 |
| 7 | Inventory 반영 |
| 8 | Docker 및 Docker Compose 설치 |
| 9 | Custom WordPress + MariaDB 배포 |
| 10 | HAProxy Reverse Proxy 구성 |
| 11 | Prometheus / Grafana / exporter 구성 |
| 12 | Health Check / Backup / Restore 검증 |

### 10.3 Playbook 구조 초안

향후 B+ 자동화 확장 시 Ansible 구조는 다음과 같이 분리한다.

~~~text
ansible/
├── ansible.cfg
├── inventory.ini
├── group_vars/
│   └── all.yml
├── playbooks/
│   ├── 01_openstack_provision.yml
│   ├── 02_wait_for_nodes.yml
│   ├── 03_configure_nodes.yml
│   ├── 04_deploy_wordpress.yml
│   ├── 05_deploy_proxy.yml
│   ├── 06_deploy_monitoring.yml
│   └── 07_validate.yml
└── site.yml
~~~

### 10.4 진행 기준

B+ Stretch Goal은 필수 구현 범위가 아니라, A급과 B급 주요 항목이 완료된 이후 진행하는 확장 목표로 둔다.

| 구분 | 기준 |
|---|---|
| 필수 여부 | 필수 아님 |
| 진행 시점 | A급 완료 및 B급 주요 항목 완료 이후 |
| 실패 대응 | 기존 A급/B급 산출물 기준으로 발표 |
| 발표 의미 | 인프라 생성부터 서비스 배포까지 End-to-End 자동화 가능성 제시 |

금일 기준으로 B+ Stretch Goal은 범위 통제 문서와 발표자료에는 “추가 확장 목표”로 반영하고, A급/B급 완료 전까지는 필수 작업으로 배정하지 않는다.

## 11. 추가 작업: 백서빈 Ubuntu 인스턴스 구성 작업 반영

Cloud Infrastructure 담당자인 백서빈의 작업 범위에 Ubuntu 기반 인스턴스 구성 작업을 추가하였다.

| 작업 항목 | 담당자 | 확인 기준 |
|---|---|---|
| Ubuntu 이미지 확인 | 백서빈 | OpenStack image list 캡처 |
| Ubuntu 인스턴스 생성 | 백서빈 | openstack server create 또는 Horizon 생성 캡처 |
| 인스턴스 ACTIVE 상태 확인 | 백서빈 | server list 캡처 |
| Floating IP 연결 | 백서빈 | FIP 연결 캡처 |
| SSH 접속 확인 | 백서빈 | ssh 접속 성공 캡처 |
