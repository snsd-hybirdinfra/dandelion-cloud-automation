### 3.3 이진욱 작업 내용 정리

Server / Virtualization 담당자인 이진욱은 노드별 리소스 기준과 기본 패키지 설치 절차를 정리하였다.

정리된 리소스 기준은 다음과 같다.

| Node | vCPU | Disk | RAM |
|---|---:|---:|---:|
| control | 2 Core | 20 GB | 2 GB |
| haproxy | 1 Core | 15 GB | 2 GB |

| Resource | Total |

### 3.4 sudoers 및 공통 패키지 설치 기준 정리

모든 노드에 대해 user1 계정의 sudo 권한을 설정하고, 기본 운영 패키지를 설치하는 절차를 정리하였다.

~~~bash
echo "user1 ALL=(ALL) NOPASSWD: ALL" | sudo tee -a /etc/sudoers
sudo apt update && sudo apt install -y vim curl wget
~~~

해당 작업은 Ansible 자동화 전 기본 접속 및 패키지 설치 검증에 활용한다.

### 3.5 Control Node 환경 구성 기준 정리

Control Node는 Ansible 실행 및 전체 노드 관리 기준 노드로 사용한다.

Control Node에는 Ansible을 설치한다.

~~~bash
sudo apt-add-repository --yes --update ppa:ansible/ansible
sudo apt update && sudo apt install -y ansible
~~~

Control Node의 주요 역할은 다음과 같다.

| 항목 | 내용 |
|---|---|
| 관리 주체 | Control Node |
| 도구 | Ansible |
| 관리 방식 | SSH / Ansible |
| 관리 대상 | Proxy, Web, DB, Backup Node |

### 3.6 HAProxy Node 환경 구성 기준 정리

HAProxy Node는 최종 기준상 HAProxy Container를 실행하는 Proxy Node로 사용한다.

따라서 Docker Engine 및 Docker Compose Plugin 설치가 필요하다.

~~~bash
sudo apt remove $(dpkg --get-selections docker.io docker-compose docker-compose-v2 docker-doc podman-docker containerd runc | cut -f1)

sudo apt update
sudo apt install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

sudo tee /etc/apt/sources.list.d/docker.sources <<EOF
Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
Components: stable
Architectures: $(dpkg --print-architecture)
Signed-By: /etc/apt/keyrings/docker.asc
EOF

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
sudo systemctl start docker
sudo systemctl enable docker
~~~

OS 패키지 기반 HAProxy 설치 방식도 검토되었으나, 최종 프로젝트 기준은 HAProxy Container 방식으로 유지한다.

### 3.7 Web Node 환경 구성 기준 정리

Web Node는 Custom WordPress Container를 실행하는 서비스 노드이다.

Web Node에도 HAProxy Node와 동일하게 Docker Engine 및 Docker Compose Plugin을 설치한다.

Web Node 구성 기준은 다음과 같다.

| 항목 | 내용 |
|---|---|
| 서비스 | Custom WordPress Container |
| Base Image | wordpress:php8.2-apache |
| 서비스 포트 | HTTP 80 |
| DB 연결 | DB Node MariaDB Service TCP 3306 |
| 접근 방식 | HAProxy Node를 통해 접근 |

---

### 3.8 DB Node 환경 구성 기준 정리

DB Node는 MariaDB를 컨테이너로 실행하지 않고 Ubuntu 패키지로 직접 설치하여 systemd 서비스로 운영한다.

~~~bash
sudo apt update && sudo apt install -y mariadb-server
sudo systemctl start mariadb
sudo systemctl enable mariadb
~~~

DB Node 구성 기준은 다음과 같다.

| 항목 | 내용 |
|---|---|
| DB Service | MariaDB |
| 설치 방식 | apt 기반 직접 설치 |
| 서비스 관리 | systemctl mariadb |
| 접근 포트 | TCP 3306 |
| 접근 허용 | Web Node, Backup Node |
| 외부 공개 | 금지 |

---

## 8. 작업 결과 요약

금일 작업을 통해 OpenStack 보안그룹 및 서브넷 생성 현황을 확인하고, 노드별 기본 환경 구성 기준을 정리하였다.

백서빈은 보안그룹 및 서브넷 생성을 진행하였고, 4개 중 3개 영역은 기본 완료 상태이다. Backup / Validation 관련 서브넷 또는 인스턴스 생성 문제는 조치 중이다.

이진욱은 Control, HAProxy, Web, DB, Backup Node의 리소스 기준과 노드별 기본 패키지 설치 절차를 정리하였다.

HAProxy Node는 안정성을 고려하여 RAM을 2 GB로 조정하였다. 이에 따라 Phase 1 기본 구성 총 리소스는 vCPU 8 Core, Disk 155 GB, RAM 12 GB로 수정한다.

Control Node에는 Ansible을 설치하고, HAProxy Node와 Web Node에는 Docker Engine 및 Docker Compose Plugin을 설치한다. DB Node에는 mariadb-server를 설치하고, Backup Node에는 mariadb-client를 설치한다.
