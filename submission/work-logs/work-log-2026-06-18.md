## 3. 수행 작업

### 3.1 최신 리소스 산정표 확정 및 인스턴스 생성

기존 리소스 산정표를 실제 구성 기준에 맞게 수정하고, 해당 기준으로 OpenStack 인스턴스 생성을 진행하였다.

최신 리소스 산정 기준은 다음과 같다.

| Node | vCPU | Disk | RAM |
|---|---:|---:|---:|
| control | 2 Core | 20 GB | 2 GB |
| haproxy | 1 Core | 15 GB | 1 GB |
| web | 1 Core | 20 GB | 2 GB |
| db | 2 Core | 20 GB | 4 GB |
| backup | 1 Core | 20 GB + 80 GB Cinder | 2 GB |
| monitoring | 1 Core | 15 GB | 2 GB |

해당 리소스 기준을 바탕으로 OpenStack 인스턴스 생성 작업을 진행하였다.

---

### 3.2 리소스 총합 재계산

최신 리소스 기준으로 전체 필요 리소스를 다시 계산하고, 해당 기준에 맞춰 인스턴스 생성 작업을 진행하였다.

| Resource | Total |
|---|---:|
| vCPU | 8 Core |
| VM Disk | 110 GB |
| Cinder Volume | 80 GB |
| Total Disk | 190 GB |
| RAM | 13 GB |

Backup Node의 80 GB Cinder Volume은 VM Root Disk와 분리하여 계산하였다.

---

### 3.3 Backup Node Cinder Volume 반영

Backup Node는 기본 OS Disk 20 GB와 별도 Cinder Volume 80 GB를 사용하는 구조로 변경하였다.

| 항목 | 기준 |
|---|---|
| Backup Node OS Disk | 20 GB |
| Cinder Volume | 80 GB |
| Mount Path | /backup |
| 저장 대상 | MariaDB dump, WordPress files archive, restore 검증 자료 |
| 사용 범위 | Backup / Validation |
| 제외 범위 | DB 원본 저장소 |

Cinder Volume은 DB Node의 원본 데이터 저장소가 아니라, Backup / Validation Node에서 백업 결과물을 보관하는 저장소로 사용한다.

---

### 3.4 Monitoring Node 추가

Monitoring Node는 Phase 2 운영 확장 범위로 추가하였다.

| 항목 | 내용 |
|---|---|
| Node | monitoring |
| vCPU | 1 Core |
| Disk | 15 GB |
| RAM | 2 GB |
| 주요 역할 | Prometheus / Grafana 운영 |
| Phase | Phase 2 |

Monitoring Node는 Prometheus와 Grafana를 통해 서비스 및 인프라 상태를 시각화하는 역할을 수행한다.

node_exporter와 cAdvisor는 각 노드의 메트릭 수집을 위해 연계할 수 있다.

---

### 3.5 포트포워딩 기반 Control Node SSH 접속 검증

최신 리소스표 기준으로 생성한 Control Node에 대해 공유기 포트포워딩을 통한 외부 PC SSH 접속이 정상적으로 수행되는 것을 확인하였다.

해당 검증을 통해 Control Node가 프로젝트의 관리 진입점 역할을 수행할 수 있음을 확인하였다.

검증된 관리 접근 구조는 다음과 같다.

~~~text
Admin
→ Router Port Forwarding
→ Control Node SSH
→ Ansible / SSH
→ Managed Nodes
~~~

관리자는 개별 노드에 직접 접속하지 않고 Control Node에만 SSH로 접근한다.

후속 작업으로 Control Node에서 각 관리 대상 노드로 SSH 접속 및 ansible ping 검증을 진행할 예정이다.

---

### 3.6 현재 구현 범위 및 Post-Phase 확장 방향 정리

현재 구현 범위에서는 단일 DB Node 구조를 유지한다.

DB Node는 MariaDB를 직접 설치하고 systemd 서비스로 운영한다.

DB Node 이중화는 현재 Phase 1~3 구현 범위에 포함하지 않고, Post-Phase 확장 방향으로 정리한다.

| 구분 | 현재 구현 | 향후 확장 |
|---|---|---|
| DB Node | 단일 DB Node | DB Node 이중화 |
| DB Service | MariaDB systemd Service | MariaDB Replication |
| 장애 대응 | Backup / Restore 검증 | DB Failover 검토 |
| 문서 반영 | 현재 구현 범위 | Post-Phase Extension |

## 8. 작업 결과 요약

금일 작업을 통해 최신 OpenStack 인스턴스 리소스 산정 기준을 확정하고, 해당 기준으로 OpenStack 인스턴스 생성을 진행하였다.

Control, HAProxy, Web, DB, Backup, Monitoring Node 기준으로 리소스를 다시 산정하고 인스턴스 생성을 진행하였으며, 기본 구성 총합은 vCPU 8 Core, VM Disk 110 GB, Cinder Volume 80 GB, Total Disk 190 GB, RAM 13 GB로 정리하였다.

Backup Node는 OS Disk 20 GB와 80 GB Cinder Volume을 분리하여 구성하고, Cinder Volume은 DB 원본 저장소가 아닌 백업 저장소로 사용하기로 하였다.

또한 공유기 포트포워딩을 통해 외부 PC에서 Control Node로 SSH 접속되는 것을 확인하였다.

이를 통해 Control Node가 프로젝트의 관리 진입점 역할을 수행할 수 있음을 확인하였으며, 후속 작업으로 Control Node에서 각 노드로 SSH 접속 및 Ansible ping 검증을 진행할 예정이다.
